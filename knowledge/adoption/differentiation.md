# BOS-FS Differentiation

> 来源: 综合 (Product Management in Practice + Continuous Discovery Habits)
> 映射: BOS-FS 五层架构 → Adoption Layer
> 用途: 明确 BOS-FS 与竞品的差异化定位，指导用户理解 Submission Engineering Runtime 的独特价值主张。

## Core Position
BOS-FS 不是"又一个 AI IDE"，而是 **Submission Engineering Runtime** — 关注"成果被接受"，而非"成果生成"。

## Competitor Analysis
| 产品 | 定位 | 核心能力 | 缺失 |
|------|------|---------|------|
| README.so | README 生成器 | 模板/Badge | 不理解项目价值 |
| Mintlify | 代码→文档 | 自动文档 | 不做审核模拟 |
| SonarQube | 代码质量 | 可维护性/安全 | 不改文档 |
| Linear | 团队管理 | Issue→Delivery | 不触及产品包装 |
| OpenHands | 自动开发 | 需求→代码→提交 | 无交付包装层 |
| Jules(Google) | 仓库理解 | 读取→理解→修改 | 不是评审系统 |
| Claude Code | 代码修改 | 仓库理解/代码修改 | 无产品包装层 |

## What Makes BOS-FS Unique
1. **统一 Skill Runtime**: 生成→表达→审核→交付串联
2. **可集成性**: 可嵌入任意 AI Agent（Cursor/Trae/Claude Code/OpenHands）
3. **Submission Engineering**: 让成果被接受
4. **评审模拟**: 内置五类评审角色（技术/投资/产品/开源/敌对）
5. **价值转换**: Feature→Capability→Outcome 公式

## HOS Ecosystem
- HOS-LS / HOS-M2F: 负责"生成"
- BOS-FS: 负责"让成果被接受"

## Target Users
个人开发者（开源发布）· 小团队（产品评审/投资路演）· 企业（交付验收/合规审核）
