# 架构质量指标体系（Architecture Quality Metrics）

> 来源: Software Architecture Metrics (Ciceri, Farley, Ford, 2023)  
> 映射: BOS-FS 五层架构 → Governance Layer + Runtime Layer  
> 用途: 提供可操作的架构质量度量方法、耦合/内聚/复杂度计算公式、技术债务量化模型和质量门限管理，直接应用于 BOS-FS Skill 系统的质量评估。

---

## 一、架构质量指标框架

### 1.1 三层指标分类

| 层级 | 指标类别 | 关注点 | 测量方式 | 更新频率 |
|------|---------|--------|---------|---------|
| **结构指标** | 模块间/模块内的结构关系 | 耦合、内聚、依赖、稳定性 | 静态代码分析 | 每次代码变更 |
| **行为指标** | 运行时的质量表现 | 响应时间、吞吐量、可用性、错误率 | 运行时观测 | 持续/按请求 |
| **演进指标** | 随时间变化的质量趋势 | 技术债务、变更成本、回归率、知识流失 | 历史数据追踪 | 每次迭代/版本 |

### 1.2 指标选择原则

**原则 1 — 可行动性（Actionable）**: 每个指标必须对应明确的改进行动。

| 指标 | 指标变差时的行动 |
|------|-----------------|
| Afferent Coupling ↑ | 减少被依赖数量，拆分接口 |
| Efferent Coupling ↑ | 引入接口抽象，依赖倒置 |
| Cyclomatic Complexity ↑ | 拆分函数，提取策略模式 |
| LCOM ↑ | 将职责分配到独立类 |
| Instability → 1.0 | 增加测试覆盖，增加消费者 |

**原则 2 — 组合度量**: 单一指标易被误导，必须组合使用。

| 组合 | 为什么 |
|------|--------|
| Instability + Abstractness | 单独看无意义，组合判断是否在 Main Sequence 上 |
| Cyclomatic Complexity + NPath | 理解分支复杂度和路径爆炸风险 |
| LCOM + CBO | 内聚低+耦合高 = 需要重构的明确信号 |

**原则 3 — 上下文敏感**: 不同架构风格的阈值不同。

| 架构风格 | 耦合容忍度 | 复杂度容忍度 |
|---------|-----------|-------------|
| 微内核（BOS-FS 当前） | 低（插件边界清晰） | 中（核心可适度复杂） |
| 微服务 | 低（网络调用代价高） | 低（服务内保持简单） |
| 事件驱动 | 中（通过事件解耦） | 中（流程编排可复杂） |

### 1.3 BOS-FS 指标采集架构

```

│  ┌──────────┐    ┌──────────┐    ┌──────────────┐   │
│  │ 源代码    │───▶│ 静态分析  │───▶│ 结构指标     │   │
│  │ / Markdown│    │ (AST)   │    │ 库内/间耦合  │   │
│  └──────────┘    └──────────┘    └──────────────┘   │

│                                                      │
│  ┌──────────┐    ┌──────────┐    ┌──────────────┐   │
│  │ Git 历史  │───▶│ 趋势分析  │───▶│ 演进指标     │   │
│  │ 变更日志  │    │ (时间序列)│    │ 债务/回归    │   │
```

**BOS-FS 实现路径**:
1. 使用 `radon` (Python 复杂度分析) 获取 Python 引擎模块的圈复杂度
2. 自定义脚本分析 Markdown Skill 文件之间的引用关系（耦合度）
3. Pipeline 运行日志自动计算行为指标
4. 每次提交触发指标快照，存入 `output/metrics/` 目录

## 二、耦合度测量体系

### 2.1 耦合度量基础定义

| 指标 | 符号 | 定义 | 计算方式 |
|------|------|------|---------|
| **Afferent Coupling** | Ca | 有多少其他模块依赖本模块 | 计算指向本模块的入度 |
| **Efferent Coupling** | Ce | 本模块依赖多少其他模块 | 计算本模块指向外的出度 |
| **Abstractness** | A | 抽象类占总类的比例 | `A = abstract_classes / total_classes` |
| **Instability** | I | 不稳定性 = Ce / (Ca + Ce) | `I = Ce / (Ca + Ce)`，范围 [0, 1] |
| **Distance from Main Sequence** | D | 偏离理想平衡的程度 | `D = \|A + I - 1\|`，范围 [0, 1] |

### 2.2 Afferent/Efferent Coupling 详解

**Afferent Coupling (Ca) — 被依赖度**:

| Ca 值 | 含义 | 风险 |
|-------|------|------|
| Ca = 0 | 无模块依赖此模块 | 可能是死代码，或独立的工具 |
| Ca = 1-2 | 少数模块依赖 | 合理，影响面可控 |
| Ca = 3-5 | 中等依赖 | 变更时需谨慎评估影响 |
| Ca > 5 | 高度被依赖 | 变更风险极高，需稳定接口+充分测试 |

**Efferent Coupling (Ce) — 主动依赖度**:

| Ce 值 | 含义 | 风险 |
|-------|------|------|
| Ce = 0 | 无外部依赖 | 可能是纯工具类，或 God Object |
| Ce = 1-3 | 少量依赖 | 合理范围 |
| Ce = 4-6 | 中等依赖 | 考虑是否引入接口抽象 |
| Ce > 6 | 过度依赖 | 高耦合风险，变更代价大 |

### 2.3 Instability (I) 详解

```
I = Ce / (Ca + Ce)

I = 0.0  ← 最稳定（只能被依赖，不依赖其他）← Stable
I = 1.0  ← 最不稳定（只依赖其他，不被依赖）← Unstable
```

| I 值 | 模块类型 | 应该放什么代码 | BOS-FS 示例 |
|------|---------|--------------|------------|
| I ≈ 0.0 | 稳定模块 | 接口定义、抽象类、常量 | `base_context.md`、Skill 协议定义 |
| I ≈ 0.2-0.4 | 相对稳定 | 核心业务逻辑 | `engine/core/` 模块 |
| I ≈ 0.6-0.8 | 相对不稳定 | 具体实现、配置 | Pipeline Stage 实现 |
| I ≈ 1.0 | 不稳定模块 | 入口、配置、启动脚本 | `main.py`、CLI 入口 |

**规则**: 依赖方向应该从 **不稳定** 指向 **稳定**（I 大的依赖 I 小的）。

### 2.4 Main Sequence 与 Distance (D)

Main Sequence 是理想状态线: `A + I = 1`，即抽象度与不稳定性互补。

```
A (Abstractness)
1.0 │  ╱
    │ ╱  ← Main Sequence (理想)
    │╱
    ╱│
  ╱ │
 ╱  │
0.0 └─────────────── I (Instability)
   0.0             1.0

D = |A + I - 1|
D = 0  → 在 Main Sequence 上（理想）
D > 0.1 → 需要关注
D > 0.3 → 架构告警：要么是"痛苦区"（过度抽象），要么是"无用区"（缺乏抽象）
```

| D 值区间 | 状态 | 含义 | 行动 |
|---------|------|------|------|
| D < 0.1 | 良好 | 抽象度与依赖度平衡 | 保持 |
| D = 0.1-0.2 | 注意 | 轻微偏离 | 观察趋势 |
| D = 0.2-0.3 | 告警 | 显著偏离 | 分析原因，制定优化计划 |
| D > 0.3 | 危险 | 严重失衡 | 立即重构 |

**痛苦区 (Pain Zone)**: A 低 + I 低 → 具体代码被大量依赖，变更代价极高。  
**无用区 (Useless Zone)**: A 高 + I 高 → 大量抽象但被很少使用，过度设计。

### 2.5 BOS-FS 耦合度实测分析

```
  └─▶ PipelineCore (Ce=2, Ca=1, I=0.67, A=0.1, D=0.23)
       └─▶ PipelineContext (Ce=0, Ca=3, I=0.0, A=0.3, D=0.3) ⚠️
       └─▶ StageProtocol (Ce=0, Ca=2, I=0.0, A=0.8, D=0.2)

各 Stage (Ce=1, Ca=0, I=1.0, A=0.0, D=0.0)

```

| 模块 | Ca | Ce | I | A | D | 状态 |
|------|----|----|---|---|---|------|

## 三、内聚度测量体系

### 3.1 内聚度量方法对比

| 指标 | 全称 | 范围 | 含义 | 计算复杂度 | 适用场景 |
|------|------|------|------|-----------|---------|
| **LCOM** | Lack of Cohesion in Methods | [0, +∞) | 不共享实例变量的方法对数 | O(n²) | Java/C++ 类分析 |
| **LCOM4** | LCOM with inheritance | [1, +∞) | 基于图连通分量的内聚度量 | O(n+e) | 考虑继承关系 |
| **TCC** | Tight Class Cohesion | [0, 1] | 直接关联的公开方法对比例 | O(n²) | 接口设计评估 |
| **NCC** | Normalized Class Cohesion | [0, 1] | 连通/直接+间接关联方法对比例 | O(n+e) | 间接关系考虑 |
| **CBO** | Coupling Between Objects | [0, +∞) | 与其他类耦合的数量 | O(n) | 类间耦合评估 |

### 3.2 LCOM (Lack of Cohesion in Methods)

```
LCOM = |{(M_i, M_j) | M_i ∩ M_j = ∅}| - |{(M_i, M_j) | M_i ∩ M_j ≠ ∅}|
       不共享实例变量的方法对数               共享实例变量的方法对数

LCOM = 0 → 完美内聚
LCOM > 0 → 内聚不足（值越大越严重）
```

**实操解读**:

| LCOM 值 | 解读 | 行动 |
|---------|------|------|
| LCOM = 0 | 所有方法共享至少一个实例变量 | ✅ 良好 |
| LCOM = 1-3 | 少量方法对不共享变量 | ⚠️ 可接受 |
| LCOM = 4-10 | 内聚不足 | 🔴 考虑拆分模块 |
| LCOM > 10 | 严重缺乏内聚 | 🔴 立即重构 |

**Python 适配**: Python 无严格"实例变量"概念，使用以下替代:
- 共享 `self.xxx` 属性访问的方法视为"共享"
- 共享同一 `PipelineContext` 字段的方法视为"共享"
- 使用 AST 分析 `self.*` 访问模式

### 3.3 TCC (Tight Class Cohesion)

```
TCC = NDC / NP
NDC = 直接连接的方法对数（共享方法调用或直接数据流）
NP  = 所有公开方法对的总数 = n×(n-1)/2

TCC = 1.0 → 完全内聚（每个方法都与另一个直接相关）
TCC = 0.0 → 无内聚（方法间完全独立）
```

**BOS-FS Stage 内聚度评估**:

| Stage | 公开方法数 | 直接连接对 | TCC | 评价 |
|-------|-----------|-----------|-----|------|
| Goal Refiner | 3 (validate_input, execute, _parse) | 2 | 0.67 | ✅ 高内聚 |
| Outcome Mapper | 4 (validate_input, execute, _map, _transform) | 4 | 0.67 | ✅ 高内聚 |
| Reviewer Simulator | 5 (validate, execute, _review_tech, _review_invest, _review_product) | 3 | 0.30 | ⚠️ 中内聚，可拆分为多个评审器 |
| Submission Builder | 4 (validate, execute, _build_bundle, _validate_bundle) | 5 | 0.83 | ✅ 高内聚 |

**内聚阈值参考**:

| 等级 | TCC 范围 | 含义 |
|------|---------|------|
| 优秀 | ≥ 0.7 | 方法高度关联 |
| 良好 | 0.5-0.7 | 适度关联 |
| 一般 | 0.3-0.5 | 存在拆分会益 |
| 差 | < 0.3 | 强烈建议拆分 |

### 3.4 CBO (Coupling Between Objects)

```
CBO = 与类 X 有耦合关系的其他类的数量
耦合关系: 方法调用、属性访问、参数传递、继承、实现接口
```

| CBO 值 | 风险 | 建议 |
|--------|------|------|
| CBO ≤ 5 | 低 | ✅ 可接受 |
| CBO = 6-10 | 中 | ⚠️ 关注趋势 |
| CBO > 10 | 高 | 🔴 需要降低耦合 |

**BOS-FS 模块 CBO 分析**:

| 模块 | CBO | 耦合来源 | 评估 |
|------|-----|---------|------|
| `PipelineCore` | 4 | PipelineContext, Stage, Config, Logger | ✅ 合理 |
| `ReviewerSimulator` | 3 | PipelineContext, ReviewRules, ScoreCalculator | ✅ 合理 |
| `PipelineContext` | 6 | 所有 Stage + Logger + Config | ⚠️ 偏高但可接受（作为数据总线） |

### 3.5 内聚-耦合联合分析

```
              高内聚
               │
      ┌────────┼────────┐
      │  理想区  │  可接受区 │
      │ 低耦合   │ 高耦合   │
      │ (保持)  │ (监控)   │
──────┼────────┼────────┼──────
      │  关注区  │  重构区   │
      │ 低内聚   │ 低内聚   │
      │ 低耦合   │ 高耦合   │
      └────────┼────────┘
               │
              低内聚
```

| 区域 | 特征 | BOS-FS 模块分布 | 行动 |
|------|------|---------------|------|
| 理想区 | 高内聚+低耦合 | 大多数 Stage 实现 | 保持 |
| 可接受区 | 高内聚+高耦合 | PipelineContext | 引入接口层降低耦合 |
| 关注区 | 低内聚+低耦合 | (无) | — |
| 重构区 | 低内聚+高耦合 | ReviewerSimulator（5 种评审逻辑合一） | 拆分为独立评审器 |

## 四、复杂度测量体系

### 4.1 圈复杂度（Cyclomatic Complexity, CC / M）

```
M = E - N + 2P
E = 控制流图的边数
N = 节点数
P = 连通分量数（通常为 1）

简化: M = 决策点数 + 1 = (if/elif/else/for/while/catch/and/or) + 1
```

| M 值 | 风险等级 | 建议 |
|------|---------|------|
| 1-5 | 低 | ✅ 简单，易于测试 |
| 6-10 | 中 | ⚠️ 适度复杂，需要充分测试 |
| 11-20 | 高 | 🔴 应该拆分 |
| 21-50 | 极高 | 🔴 不可维护，必须重构 |
| > 50 | 危险 | 🔴 无法测试，不可信任 |

**BOS-FS 引擎圈复杂度扫描**（使用 `radon cc` 工具）:

| 模块 | 函数 | CC | 评估 |
|------|------|----|------|
| `main.py` | `main()` | 8 | ⚠️ CLI 参数处理逻辑多，可接受 |
| `main.py` | `run_pipeline()` | 12 | 🔴 异常处理+状态分支过多，需拆分 |
| `book_ingestor.py` | `process_book()` | 15 | 🔴 需要拆分为解析/分析/存储子函数 |
| `pipeline_core.py` | `execute_stage()` | 6 | ⚠️ 合理 |
| `goal_refiner.py` | `_parse_intent()` | 9 | ⚠️ 意图解析逻辑复杂，可考虑策略模式 |

**测试覆盖要求**（与 CC 对应）:

| CC 范围 | 最少测试用例数 | 说明 |
|---------|--------------|------|
| 1-5 | 1-2 | 基础路径覆盖 |
| 6-10 | 3-5 | 每个决策分支至少一个用例 |
| 11-20 | 6-10 | 需要边界值+等价类 |
| > 20 | 10+ | 必须重构降低复杂度 |

### 4.2 Halstead 复杂度

| 指标 | 符号 | 定义 | 含义 |
|------|------|------|------|
| 操作符数量 | n₁ | 唯一操作符数量 | 词汇多样性 |
| 操作数数量 | n₂ | 唯一操作数数量 | 数据多样性 |
| 总操作符 | N₁ | 操作符出现总次数 | 程序长度一部分 |
| 总操作数 | N₂ | 操作数出现总次数 | 程序长度一部分 |
| 程序长度 | N = N₁ + N₂ | 总 token 数 | 代码规模 |
| 词汇表 | n = n₁ + n₂ | 唯一 token 数 | 代码多样性 |
| 体积 | V = N × log₂(n) | 程序信息量 | 实现工作量 |
| 难度 | D = (n₁/2) × (N₂/n₂) | 理解难度 | 值越大越难理解 |
| 工作量 | E = D × V | 编写难度 | 与 V 成正比，与 D 成正比 |
| 理解时间 | T = E / 18 秒 | 预估理解时间 | 超过 30 分钟 = 需要重构 |

**Halstead 阈值参考**:

| 指标 | 健康 | 关注 | 告警 |
|------|------|------|------|
| 体积 V | < 5000 | 5000-15000 | > 15000 |
| 难度 D | < 15 | 15-30 | > 30 |
| 工作量 E | < 100000 | 100K-500K | > 500K |
| 理解时间 T | < 30min | 30min-2h | > 2h |

### 4.3 维护指数（Maintainability Index, MI）

```
MI = max(0, (171 - 5.2 × ln(V) - 0.23 × CC - 16.2 × ln(LOC)) × 100 / 171)

其中:
V = Halstead 体积
CC = 圈复杂度
LOC = 代码行数
```

| MI 范围 | 等级 | 颜色 | 含义 |
|---------|------|------|------|
| 85-100 | 高可维护性 | 🟢 | 易于理解和修改 |
| 65-84 | 中等可维护性 | 🟡 | 适度可维护，需要注意 |
| 0-64 | 低可维护性 | 🔴 | 难以维护，建议重构 |

### 4.4 NPath 复杂度（路径爆炸）

```
NPath = 函数中所有可能的执行路径数量
```

| NPath 值 | 风险 |
|----------|------|
| < 200 | ✅ 可接受 |
| 200-1000 | ⚠️ 需要简化 |
| > 1000 | 🔴 路径爆炸，无法充分测试 |

**降低 NPath 的策略**:
1. 用多态/策略模式替代 if-else 链
2. 用表驱动方法替代嵌套条件
3. 提前返回（Guard Clause）减少嵌套
4. 将嵌套逻辑提取为独立函数

## 五、架构技术债务量化

### 5.1 技术债务分类框架

| 债务类型 | 定义 | 量化方法 | 利息成本 |
|---------|------|---------|---------|
| **设计债务** | 架构决策不当导致修改困难 | 耦合度超标模块数 × 修复成本 | 每次变更多花 30-50% 时间 |
| **代码债务** | 复杂度高、可读性差的代码 | 圈复杂度超标函数数 | Bug 率高 2-3 倍 |
| **测试债务** | 缺少自动化测试 | 覆盖率缺口 × 函数数 | 每次修改需手动验证 |
| **文档债务** | 架构文档与实际不符 | ADR 缺失数 + 过时文档数 | 新人上手时间增加 |
| **依赖债务** | 过时/不安全的第三方依赖 | 过期依赖数 × 升级难度 | 安全风险 + 兼容性风险 |
| **知识债务** | 关键知识仅在个人脑中 | 单点知识持有者数量 | 人员流失 = 知识流失 |

### 5.2 SQALE 技术债务计算方法

```
技术债务比率 (TD Ratio) = 修复成本 / 开发成本

修复成本 = Σ(每个违规项的修复时间)
开发成本 = 当前功能的总开发时间

TD Ratio 解释:
  < 5%  → 债务可控
  5-10% → 需要关注
  10-20% → 债务严重，影响生产力
  > 20% → 债务危机，需要专项偿还
```

**BOS-FS 技术债务估算**:

| 债务项 | 违规数 | 修复时间/项 | 总修复时间 | 优先级 |
|--------|--------|------------|-----------|--------|
| 圈复杂度 > 10 | 2 | 4h | 8h | 🔴 P0 |
| TCC < 0.3 | 1 | 8h | 8h | 🔴 P0 |
| D > 0.2 | 2 | 6h | 12h | 🟡 P1 |
| 缺少架构测试 | 3 | 4h | 12h | 🟡 P1 |
| 知识文件同步 | 5 | 1h | 5h | 🟢 P2 |
| **总计** | | | **45h** | |

### 5.3 技术债务利息模型

```
债务利息 = 每次变更额外耗时 × 变更频率

示例:
  PipelineContext 接口缺失 → 每次新增 Stage 多花 1h
  每周新增/修改 2 个 Stage
  周利息 = 2h
  月利息 = 8h
  年利息 = 96h（约 2.5 人周）

偿还收益 = 修复成本 / 年利息 = 12h / 96h = 8× ROI
```

**高 ROI 债务优先偿还**:

| 债务项 | 修复成本 | 年利息 | ROI | 优先级 |
|--------|---------|--------|-----|--------|
| PipelineContext 接口抽象 | 12h | 96h | 8× | 🔴 |
| ReviewerSimulator 拆分 | 8h | 48h | 6× | 🔴 |
| Stage 接口契约测试 | 12h | 36h | 3× | 🟡 |

```
┌────────────────────────────────────────────────────┐
│  BOS-FS 技术债务看板                                 │
├──────────────┬──────┬──────┬──────┬────────────────┤
│ 债务项        │ 本金  │ 月利息│ ROI  │ 状态           │
├──────────────┼──────┼──────┼──────┼────────────────┤
│ PC 接口抽象   │ 12h  │ 8h   │ 8×  │ 📋 计划中       │
```

### 6.1 质量门限分级

| 级别 | 门限名称 | 触发条件 | 后果 |
|------|---------|---------|------|
| **硬门限** | 必须满足 | 违反则 CI 失败 | 不允许合并 |
| **软门限** | 应该满足 | 违反则告警 | 需要评审批准 |
| **趋势门限** | 不应恶化 | 指标变差 | 需要说明原因 |
| **信息门限** | 仅记录 | 无阈值 | 趋势追踪 |

### 6.2 BOS-FS 质量门限定义

#### 硬门限（CI Gate）

| 指标 | 阈值 | 测量工具 | 执行点 |
|------|------|---------|--------|
| Schema 验证通过率 | 100% | JSON Schema 校验 | 每次 Pipeline 运行 |
| 测试覆盖率（新增代码） | ≥ 80% | pytest-cov | CI Pipeline |
| 单函数 CC 上限 | ≤ 25 | radon | 代码提交检查 |
| 知识文件一致性 | 无引用断裂 | 自定义校验脚本 | 代码合并前 |

#### 软门限（Review Gate）

| 指标 | 阈值 | 测量方式 | 执行点 |
|------|------|---------|--------|
| 平均圈复杂度 | ≤ 10 | radon | 每周质量报告 |
| 模块平均 TCC | ≥ 0.5 | 自定义分析 | 每周质量报告 |
| 最大 D 值 | ≤ 0.3 | 依赖分析 | 架构评审 |
| MI 平均值 | ≥ 65 | radon mi | 每月质量报告 |

#### 趋势门限

| 指标 | 规则 | 监控方式 |
|------|------|---------|
| CC 趋势 | 不得连续 3 次迭代上升 | 周报告趋势图 |
| 技术债务比率 | 不得上升超过 2% | 版本对比 |
| 耦合度 | 模块间依赖不得增加 | 依赖图 diff |

## 逐章深度提炼 v0.2.0（基于原文逐章读取，NEED4 重新校准）

### Chapter 1: What Are Architecture Metrics?
**核心论点**: 架构度量是量化评估架构质量的手段。与代码度量不同，架构度量关注模块间关系、系统结构和演进趋势。
**关键概念**: 结构度量/行为度量/演进度量三类、度量可行动性原则、组合度量、上下文敏感阈值
**实践方法**: 每个度量对应明确改进行动；单一度量易被误导需组合使用；不同架构风格有不同阈值
**BOS-FS 映射**: Pipeline 四类评审报告即"架构质量度量"——技术评审测代码质量、投资评审测投资回报、产品评审测产品适配、开源评审测供应链风险。

### Chapter 2: Modularity Metrics
**核心论点**: 模块化质量通过耦合和内聚度量。好架构有低耦合（模块间少依赖）和高内聚（模块内强关联）。
**关键概念**: Afferent Coupling (Ca)、Efferent Coupling (Ce)、Instability (I = Ce/(Ca+Ce))、Abstractness (A)、Main Sequence
**实践方法**: Ca↑ 表示被广泛依赖，Ce↑ 表示依赖太多外部模块；I=0 最稳定，I=1 最不稳定；模块应在 Main Sequence 附近（A+I≈1）
**BOS-FS 映射**: BOS-FS Skill 间耦合度量——每个 Skill 的 Ca/Ce 可通过 import 分析计算。Pipeline 可检测"过度耦合的 Skill"。Stage 间通过 JSON 数据总线通信，耦合度低。

### Chapter 3: Dependency Metrics
**核心论点**: 依赖度量揭示架构中的"坏味道"——循环依赖、上帝组件、不稳定依赖。依赖图是理解架构的利器。
**关键概念**: 循环依赖检测、God Component（过度连接组件）、Stable Dependencies Principle（SDP）、依赖注入
**实践方法**: 用依赖图可视化架构；消除循环依赖；确保依赖指向稳定方向；限制单个组件的依赖数量
**BOS-FS 映射**: BOS-FS Stage 依赖图应无循环依赖——Goal Refiner→Outcome Mapper→Reviewer→Submission Builder 是单向流。Pipeline 可检测依赖违规。

### Chapter 4: Maintainability Metrics
**核心论点**: 可维护性度量预测代码修改的难易程度。高可维护性架构支持快速变更和低回归风险。
**关键概念**: Cyclomatic Complexity、NPath Complexity、Maintainability Index、代码气味检测
**实践方法**: 控制函数圈复杂度（<10）；用 NPath 检测路径爆炸；定期计算 Maintainability Index 追踪趋势
**BOS-FS 映射**: Pipeline 技术评审应检查代码可维护性指标。Pipeline 自身的代码也应定期测量可维护性，确保长期可演进。

### Chapter 5: Technical Debt Metrics
**核心论点**: 技术债务是可量化的——通过修复成本、影响范围和业务风险来计算。管理技术债务需要持续的度量和优先级排序。
**关键概念**: 技术债务比率（修复成本/总开发成本）、债务利息（额外维护成本）、债务本金（一次性修复成本）、债务热力图
**实践方法**: 定期扫描代码库计算债务；按"利息×影响范围"排序修复优先级；建立债务预算（每迭代分配 20% 容量还债）
**BOS-FS 映射**: Pipeline 技术评审检测技术债务——代码异味、重复代码、过度复杂等。投资评审评估债务修复的 ROI。Pipeline 报告是债务跟踪工具。

### Chapter 6: Testing Metrics
**核心论点**: 测试度量不仅仅是代码覆盖率——还需要测试质量、测试速度、测试可维护性等多维度指标。
**关键概念**: 代码覆盖率、分支覆盖率、变异测试分数、测试执行时间、测试稳定性（flaky test 比例）
**实践方法**: 覆盖率>80% 是基线但不是目标；变异测试评估测试质量；消除 flaky test；监控测试执行时间趋势
**BOS-FS 映射**: Pipeline 应集成测试度量——检查测试覆盖率是否达标、变异测试分数是否合格。Pipeline 自身也需高质量测试保障。

### Chapter 7: Performance Metrics
**核心论点**: 性能度量需要从用户视角（响应时间）和系统视角（吞吐量、资源利用率）两个维度考虑。性能优化需要数据驱动。
**关键概念**: 响应时间（P50/P95/P99）、吞吐量（QPS/RPS）、资源利用率（CPU/内存/IO）、性能预算
**实践方法**: 建立性能预算（每页加载<3秒）；P99 比平均值更重要；性能回归自动检测；容量规划基于历史趋势
**BOS-FS 映射**: Pipeline 执行性能度量——各 Stage 耗时、总执行时间、Token 消耗成本。Pipeline 应建立性能预算和回归检测。

### Chapter 8: Security Metrics
**核心论点**: 安全度量从"漏洞数量"转向"风险暴露面"和"修复速度"。安全需要左移——在开发早期检测而非部署后修复。
**关键概念**: 漏洞密度（漏洞数/KLOC）、平均修复时间（MTTR）、漏洞逃逸率（生产发现的漏洞/总漏洞）、安全左移指标
**实践方法**: 在 CI/CD 中集成安全扫描；跟踪漏洞修复 SLA；计算漏洞逃逸率降低趋势；定期红队测试
**BOS-FS 映射**: Pipeline 技术评审和开源评审是"安全左移"工具——在提交阶段检测安全漏洞和供应链风险。Pipeline 是安全度量的数据采集点。

### Chapter 9: Architecture Quality Gates
**核心论点**: 质量门是架构治理的执行机制——在关键节点（合并/部署/发布）自动检查架构质量指标。不合格则阻止流程继续。
**关键概念**: 质量门定义、自动化检查、质量门级别（Warning/Error/Block）、质量门演进
**实践方法**: 定义关键质量指标阈值；在 CI/CD 中集成自动检查；区分 Warning（建议）和 Block（必须）；定期回顾阈值合理性
**BOS-FS 映射**: BOS-FS Pipeline 本身即"架构质量门"——四类评审是四个质量门，每个门有通过/失败标准。Pipeline 阻止低质量提交进入生产。

### Chapter 10: Architecture Dashboard
**核心论点**: 架构仪表盘是架构质量的可视化呈现——将多维指标整合为可操作的视图，支持持续监控和趋势分析。
**关键概念**: 仪表盘设计原则、指标聚合、趋势分析、告警阈值、可操作视图
**实践方法**: 仪表盘面向受众定制（架构师/开发/管理层）；趋势分析比单点值更重要；告警基于异常检测而非固定阈值
**BOS-FS 映射**: Pipeline 评审报告即"架构仪表盘"——每个维度（技术/投资/产品/开源）有独立视图，汇总为整体质量评分。报告支持趋势分析。
