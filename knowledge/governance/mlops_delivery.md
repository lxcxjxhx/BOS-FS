# MLOps 交付工程框架

> 来源: Engineering MLOps — Rapidly Build, Test, and Manage Production-Ready Machine Learning Life Cycles at Scale (Emmanuel Raj)
> 映射: BOS-FS 五层架构 → Intent Layer + Governance Layer + Runtime Layer
> 用途: 提供 MLOps 成熟度评估、ML 交付流水线架构、模型版本管理、特征工程设计、部署策略和监控框架，指导 BOS-FS 知识摄取与评估系统的工程化实践。

---

## 一、MLOps 成熟度模型（0-3级）

### 1.1 成熟度分级定义

| 等级 | 名称 | 特征 | 自动化程度 | 典型组织 | BOS-FS 映射 |
|------|------|------|-----------|---------|------------|
| **L0** | 手动 ML | 无 DevOps 实践；Jupyter Notebook → 手动部署；无版本控制；无监控 | 0% | 研究实验室、POC 阶段 | 知识摄取前的手动数据整理 |
| **L1** | ML Pipeline | 基础 CI/CD；数据版本控制；基础模型注册；手动触发训练 | 30-50% | 初创 AI 团队 | BOS-FS Pipeline 基础知识摄取 |
| **L2** | CI/CD for ML | 自动化训练-评估-部署流水线；模型版本管理；自动化监控；A/B 测试 | 50-80% | 成熟 AI 团队 | BOS-FS 完整 7 Stage Pipeline |
| **L3** | Fully Automated ML | 自动触发训练（数据/模型漂移触发）；AutoML；自动回滚；自适应监控 | > 80% | 平台化 ML 组织 | BOS-FS 未来 DAG + 事件驱动架构 |

### 1.2 成熟度评估矩阵

```
MLOps 成熟度评估 (24 项检查):

L0 → L1 关键里程碑 (8 项):
  ☐ 数据版本控制 (DVC/Git-LFS)
  ☐ 基础实验追踪 (MLflow/W&B)
  ☐ 模型序列化格式标准化
  ☐ 基础 CI Pipeline (代码 lint + 单元测试)
  ☐ 部署脚本自动化
  ☐ 基础监控（服务可用性）
  ☐ 环境一致性（容器化）
  ☐ 文档化 ML 流程

L1 → L2 关键里程碑 (8 项):
  ☐ 自动化训练 Pipeline
  ☐ 模型注册表
  ☐ 自动化评估 Pipeline
  ☐ 数据质量检查
  ☐ 模型版本与代码版本关联
  ☐ A/B 测试框架
  ☐ 模型漂移检测
  ☐ 自动化部署 Pipeline

L2 → L3 关键里程碑 (8 项):
  ☐ 漂移触发自动重训练
  ☐ AutoML 集成
  ☐ 自动回滚机制
  ☐ 多臂老虎机实验
  ☐ 自适应监控阈值
  ☐ 特征存储 (Feature Store)
  ☐ 模型可解释性自动化
  ☐ 全链路可观测性
```

### 1.3 BOS-FS MLOps 成熟度对标

| 维度 | BOS-FS 当前状态 | 对应 MLOps 等级 | 改进方向 |
|------|----------------|----------------|---------|
| 知识摄取 (book_ingestor) | 手动触发 → 自动化解析 | L1 → L2 | 漂移触发自动更新 |
| 评估框架 (reviewer_simulator) | 规则化评审 | L1 | 引入 ML 模型评审 |
| 意图解析 (goal_refiner) | 确定性规则 | L1 | 可引入 NLP 增强 |
| 交付流水线 (Pipeline) | 顺序自动化 | L1 → L2 | DAG + 并行执行 |
| 监控 (PipelineContext) | 运行日志 | L0 → L1 | 增加结构化监控 |

### 2.2 BOS-FS 知识摄取流水线映射

```
ML Pipeline Stage          ↔  BOS-FS Pipeline Stage
──────────────────────────────────────────────────────
数据获取 (Data Acquisition)  →  项目描述/代码库读取 (input/)
数据清洗 (Data Cleaning)     →  意图解析和标准化 (Goal Refiner)
特征工程 (Feature Eng)       →  价值转换和结构化 (Outcome Mapper)
模型训练 (Training)          →  知识文件生成 (Book Knowledge Ingestor)
模型验证 (Validation)        →  评审模拟 (Reviewer Simulator)
模型注册 (Registry)          →  knowledge/ 目录版本化
部署 (Deployment)            →  submission_bundle/ 生成
监控 (Monitoring)            →  Pipeline 运行日志 + rubrics.md 评分
```

### 2.3 流水线组件设计原则

| 原则 | 内容 | ML 场景 | BOS-FS 场景 |
|------|------|---------|------------|
| **幂等性** | 相同输入产生相同输出 | 确定性训练种子 | Pipeline cache_key 机制 |
| **可重现性** | 记录所有参数和版本 | 实验追踪系统 | PipelineContext 完整日志 |
| **可回滚** | 任一阶段失败可回退 | 模型版本管理 | Stage 异常隔离 + resume |
| **可监控** | 每个组件暴露指标 | Prometheus + Grafana | Pipeline 运行指标收集 |
| **可扩展** | 水平扩展计算资源 | Kubernetes + 分布式训练 | 未来 DAG 并行 Stage |

### 3.2 实验追踪框架

| 类别 | 记录内容 | 工具示例 | BOS-FS 实现 |

### 4.2 BOS-FS 特征工程映射

在 BOS-FS 中，"特征工程"映射为**知识提取与结构化**过程：

| ML 特征工程概念 | BOS-FS 对应实现 | 说明 |
|----------------|----------------|------|
| 原始数据 | 项目描述/代码库/截图 | 输入的原始材料 |
| 数据清洗 | 意图标准化 (Goal Refiner) | 去除模糊描述、标准化格式 |
| 特征提取 | 价值转换 (Outcome Mapper) | Feature→Capability→Outcome |
| 特征选择 | 评审规则筛选 (Reviewer Simulator) | 聚焦评审关注的关键信息 |
| 特征存储 | knowledge/ 目录 | 可复用的知识特征库 |
| 特征版本 | Git 标签 + 文件 hash | 确保知识版本可追溯 |

### 4.3 特征质量检查清单

| 检查项 | 标准 | 检测方法 | 修复方式 |
|--------|------|---------|---------|
| 数据完整性 | 缺失值 < 5% | 缺失率统计 | 插补/删除 |
| 数据一致性 | 无矛盾记录 | 约束检查 | 冲突解决规则 |
| 特征稳定性 | PSI < 0.1 | Population Stability Index | 重新分箱/转换 |
| 特征区分力 | IV > 0.02 | Information Value | 剔除弱特征 |
| 特征独立性 | VIF < 10 | Variance Inflation Factor | PCA/剔除共线特征 |
| 特征时效性 | 无过期特征 | 时间戳检查 | 特征淘汰策略 |

## 五、模型注册表和部署策略

### 5.1 模型注册表 (Model Registry)

**生命周期状态机**:

```
                   ┌──────────┐
                   │  None    │ ← 初始/实验
                   └────┬─────┘
                        │ 验证通过
                   ┌────▼─────┐
          ┌───────│ Staging  │───────┐
          │       └────┬─────┘       │
          │            │ 生产审批     │
          │       ┌────▼─────┐       │
          │       │ Production│       │
          │       └────┬─────┘       │
          │            │ 性能退化     │
          │       ┌────▼─────┐       │
          └──────▶│ Archived │◀──────┘
                  └──────────┘
```

**注册表操作**:
```python
class ModelRegistry:
    def register(self, model: Model, metadata: dict) -> str:
        """注册新版本模型，返回 version_id"""
    
    def promote(self, version_id: str, target_stage: str) -> bool:
        """提升模型到目标阶段 (staging → production)"""
    
    def get_production_model(self) -> Model:
        """获取当前生产模型"""
    
    def rollback(self, version_id: str) -> bool:
        """回滚到指定版本"""
    
    def list_versions(self, model_name: str) -> list:
        """列出模型所有版本"""
```

### 5.2 部署策略对比

| 策略 | 流量分配 | 风险 | 适用场景 | 回滚难度 | BOS-FS 应用 |
|------|---------|------|---------|---------|------------|
| **直接替换** | 0%→100% | 🔴 高 | 低风险变更 | 难 | 不推荐 |
| **Canary 部署** | 逐步增加 (5%→25%→50%→100%) | 🟡 中 | 大多数变更 | 易 | Pipeline Stage 灰度验证 |
| **A/B 测试** | 随机分流 (50/50) | 🟢 低 | 需要数据验证 | 易 | 评审规则 A/B 对比 |
| **Blue/Green** | 双环境切换 | 🟢 低 | 零停机要求 | 易 | 提交包版本切换 |
| **Shadow** | 100% 复制但不影响结果 | 🟢 极低 | 高风险变更验证 | 易 | 评审模拟 vs 真实评审 |

### 5.3 BOS-FS 部署策略实现

**Shadow 评审模式**（推荐）:
```
流程:
  用户提交项目
       │
       ├──▶ 真实评审人 (Blue)
       │        └── 人工评审结果
       │
       └──▶ BOS-FS Reviewer Simulator (Green/Shadow)
                └── AI 模拟评审结果
                      │
                      ▼
              对比两者一致性
              • 一致性 ≥ 80% → AI 评审可信
              • 一致性 < 80% → 需要改进评审规则
```

**Canary 知识更新**:
```
更新 knowledge/ 文件时的渐进式部署:
  1. 新知识文件写入 knowledge/draft/
  2. Pipeline 运行验证 (lint/结构检查)
  3. 小范围用户使用 (5% Pipeline 运行)
  4. 收集反馈评分 (rubrics.md)
  5. 评分达标 → 移至 knowledge/ 正式目录
  6. 评分不达标 → 回滚 + 标记改进
```

### 5.4 部署检查清单

| 序号 | 检查项 | 通过标准 | 自动检查 |
|------|--------|---------|---------|
| D1 | 模型版本已注册 | 注册表中有记录 | ✅ |
| D2 | 离线评估通过 | 所有指标 ≥ 基准 | ✅ |
| D3 | 公平性检查通过 | 无显著偏差 | ✅ |
| D4 | 性能基准通过 | 延迟/P99 达标 | ✅ |
| D5 | 回滚方案就绪 | 前一个版本可用 | ✅ |
| D6 | 监控就绪 | 告警规则已配置 | ✅ |
| D7 | 文档已更新 | 版本说明完整 | ☑ 手动 |

```
│  │  • 准确率/召回率/F1 下降                           │   │
│  │  • 特征重要性变化                                  │   │
│  │  • 预测置信度分布                                  │   │
│  └───────────────────────────────────────────────────┘   │
│  ┌───────────────────────────────────────────────────┐   │
│  │  5. 业务监控                                       │   │
│  │  • 业务指标影响 (转化率/留存率)                     │   │
│  │  • 用户反馈                                       │   │
│  │  • ROI 追踪                                       │   │
│  └───────────────────────────────────────────────────┘   │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 6.2 漂移检测方法

**数据漂移 (Data Drift)**:

| 方法 | 适用场景 | 检测目标 | 阈值 |
|------|---------|---------|------|
| **PSI** (Population Stability Index) | 数值/类别特征 | 输入分布变化 | PSI > 0.2 = 显著漂移 |
| **KS 检验** (Kolmogorov-Smirnov) | 数值特征 | 累积分布差异 | p < 0.05 = 显著漂移 |
| **卡方检验** (Chi-Square) | 类别特征 | 类别比例变化 | p < 0.05 = 显著漂移 |
| **JS 散度** (Jensen-Shannon) | 任意分布 | 分布相似度 | JS > 0.1 = 显著漂移 |

**概念漂移 (Concept Drift)**:

| 方法 | 原理 | 适用场景 |
|------|------|---------|
| **性能监控** | 准确率/损失持续下降 | 有标签场景 |
| **预测分布监控** | 预测分布显著变化 | 无标签场景 |
| **残差分析** | 预测误差模式变化 | 回归任务 |
| **Page-Hinkley 检验** | 序列数据均值漂移 | 流式数据 |

### 6.3 漂移响应策略

```
漂移检测 → 响应决策树:

漂移检测触发
    │
    ├── 数据漂移?
    │     ├── 轻微 (PSI 0.1-0.2) → 持续监控，准备重训练
    │     ├── 中等 (PSI 0.2-0.3) → 计划重训练，调查原因
    │     └── 严重 (PSI > 0.3)   → 立即重训练 / 回滚
    │
    └── 概念漂移?
          ├── 性能下降 < 5% → 持续监控
          ├── 性能下降 5-15% → 计划重训练
          └── 性能下降 > 15% → 立即回滚 + 重训练
```

### 6.4 BOS-FS 知识漂移监控

BOS-FS 中的"知识漂移"指 knowledge/ 文件与最新行业实践之间的偏差：

| 漂移类型 | 检测方式 | 响应动作 |
|---------|---------|---------|
| **框架过时** | 知识文件引用 vs 最新版本 | 更新引用 + 重摄取 |
| **实践演进** | DORA 基准年度更新 | 同步最新 State of DevOps Report |
| **工具变更** | 引用的工具链废弃 | 更新知识文件中的工具推荐 |
| **安全漏洞** | CVE 数据库扫描依赖 | 更新 requirements.txt + 知识文件 |

**漂移监控实现方案**:
```python
class KnowledgeDriftDetector:
    """检测知识文件是否需要更新"""
    
    def check_drift(self, knowledge_file: str) -> DriftReport:
        # 1. 检查引用来源是否有新版本
        # 2. 检查知识文件最后更新时间
        # 3. 检查相关行业报告是否有更新
        # 4. 计算漂移分数

            file=knowledge_file,
```
