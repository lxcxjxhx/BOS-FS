# 开源项目评审规则

> 来源: 综合 (多书评审规则整合)
> 映射: BOS-FS 五层架构 → Governance Layer
> 用途: 开源项目维度评审标准，涵盖社区价值、技术选型、可维护性等评估规则。

## 社区价值评估标准（25% 权重）

- **问题通用性**：是否解决开发者社区的共性问题，而非个别场景的特殊需求
- **独特性**：与现有方案是否有明确差异，提供独特价值或更优解
- **目标用户**：目标受众是否为开发者而非终端用户，定位清晰
- **生态贡献**：是否填补技术空白或改善现有工具链

| 扣分项 | 分值 |
|--------|------|
| 问题不通用（非开发者痛点） | -20 |
| 无独特性（与现有方案同质化） | -15 |
| 目标用户不明确 | -10 |
| 生态贡献低（不填补技术空白） | -10 |

## 文档完整性评审（25% 权重）

- **README**：是否包含项目简介、安装步骤、快速上手指南
- **API文档**：接口说明是否完整，参数类型、返回值、异常处理是否清晰
- **示例代码**：是否提供可运行的示例或 Demo 项目
- **贡献指南**：是否包含 CONTRIBUTING.md，说明如何提交 Issue/PR

| 扣分项 | 分值 |
|--------|------|
| README 不完整（缺简介/安装/使用） | -20 |
| 无 API 文档 | -15 |
| 无可运行示例代码 | -10 |
| 无贡献指南（CONTRIBUTING.md） | -10 |

## 可维护性评审标准（20% 权重）

- **代码结构**：是否模块化、职责清晰、命名规范、可读性好
- **CI/CD**：是否配置持续集成（GitHub Actions / Travis CI 等），提交自动构建
- **测试覆盖**：核心功能是否有单元测试/集成测试，覆盖率 ≥ 70%
- **版本管理**：是否遵循语义化版本（SemVer），是否有 CHANGELOG.md

| 扣分项 | 分值 |
|--------|------|
| 代码结构混乱（无模块划分/可读性差） | -20 |
| 无 CI/CD 配置 | -15 |
| 无测试覆盖 | -15 |
| 无版本管理（无 SemVer/CHANGELOG） | -10 |

## 许可证合规评审（15% 权重）

- **许可证选择**：是否在根目录包含 LICENSE 文件，类型明确（MIT/Apache/GPL 等）
- **依赖兼容**：第三方依赖的许可证是否与本项目许可证兼容
- **版权声明**：源文件是否包含 copyright header
- **第三方组件**：是否在 NOTICE 或 THIRD-PARTY 文件中声明所有第三方依赖

| 扣分项 | 分值 |
|--------|------|
| 无开源许可证 | -25 |
| 依赖许可证不兼容 | -15 |
| 无版权声明（copyright header） | -5 |
| 未声明第三方依赖 | -10 |

## 社区治理评审（15% 权重）

- **维护活跃度**：Commit 频率、Issue 响应速度、Release 周期
- **决策流程**：是否有 RFC 机制、设计文档或治理文档
- **代码审查**：是否有 PR Review 机制，代码合并需经审核
- **社区参与**：是否开放外部贡献（Issue/PR 欢迎标识、Good First Issue 标签）

| 扣分项 | 分值 |
|--------|------|
| 维护不活跃（长期无提交/不响应） | -15 |
| 无决策流程（无 RFC/设计文档） | -10 |
| 无代码审查机制 | -10 |
| 不接受外部贡献 | -5 |
