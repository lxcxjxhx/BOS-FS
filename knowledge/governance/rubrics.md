# BOS-FS 评分标准

> 来源: 综合 (Software Architecture Metrics + ISO 25010)
> 映射: BOS-FS 五层架构 → Governance Layer
> 用途: 定义 BOS-FS 流水线各阶段的评分标准、权重分配与质量门限，用于量化评估提交包质量。

## 一、流水线综合评分 (Pipeline Score)

### 阶段权重
| 阶段 | 权重 | 评分标准 |
|------|------|----------|
| Goal Refinement | 14% | persona/problem/solution/outcome 完整度 |
| Outcome Mapping | 14% | Feature→Capability→Outcome 转换质量 |
| README Refactor | 24% | 五段式结构完整度、价值公式应用 |
| Review Simulation | 24% | 四类/五类评审平均通过率 |
| Submission Build | 19% | 提交包组件完整性、一致性 |
| Trust 信任背书 | 5% | 权威引用/行业对标/透明度 |

### 阶段评分细则（均为 0-10 分）
#### Goal Refinement: 8-10四字段完整具体有量化；5-7基本完整部分泛化；2-4字段缺失/模糊；0-1无法提取意图
#### Outcome Mapping: 8-10全部Feature可量化转换；5-7主要Feature有转换；2-4仅Capability层；0-1无有效转换
#### README Refactor: 8-10五段式完整架构图清晰；5-7结构基本完整；2-4结构混乱；0-1未重构
#### Review Simulation: 8-10平均通过率>70%；5-7通过率50-70%；2-4通过率30-50%；0-1通过率<30%
#### Adversarial Review: 8-10项目无明显致命缺陷；5-7有可控风险；2-4存在重大问题；0-1多个致命缺陷
#### Submission Build: 8-10所有组件完整一致；5-7核心完整少量缺失；2-4部分生成；0-1未生成/严重不一致
#### Trust: 8-10有充分权威引用+行业对标+透明度高；5-7有基本权威引用；2-4仅表面引用无关联；0-1无任何信任背书

### 综合评分: `pipeline_score = Σ(stage_score × weight)` （0-10 分制）
| 分数 | 含义 |
|------|------|
| ≥ 9.0 | 可直接提交 |
| 8.0-8.9 | 建议微调 |
| 6.0-7.9 | 需要改进 |
| 4.0-5.9 | 大幅修改 |
| < 4.0 | 重新规划 |

> 每个评审类型返回各维度具体 0-10 分数 + 权重 + 总分，不搞抽象等级。
> 代码实现: `engine/core/06_review_simulator/reviewer_simulator.py`

### 开源评审
| 维度 | 权重 | 9-10 | 7-8 | 5-6 | 3-4 | 0-2 |
|------|------|------|------|------|------|------|
| 开源许可证 | 30% | MIT/Apache/BSD | GPL/MPL清晰 | 许可证有效 | 不常见许可证 | 缺失 |
| 文档完整度 | 25% | README+API+安装 | 主要文档齐全 | 基本文档 | 严重缺失 | 无文档 |
| 贡献指引 | 15% | CONTRIBUTING.md | 有贡献说明 | 基本指引 | 过于简略 | 缺失 |
| 行为准则 | 10% | CODE_OF_CONDUCT | 有行为准则 | 提及 | 缺失 | 无 |
| 第三方合规 | 20% | ≥3个依赖声明清晰 | 有依赖声明 | 部分声明 | 过于简略 | 缺失 |

**通过阈值**: 加权总分 ≥ 7.0/10

> **深度不足惩罚**: 任一评审类型的 7 层检查中缺失 ≥2 层时，该评审阶段分数 ×0.85。

> **敌对评审惩罚**: 当存在 adversarial 评审时，pipeline_score 额外乘以 adversarial 系数：系数 = pass_probability_adversarial / 100。即敌对评审通过率越低，综合评分越低。

### 敌对评审（Adversarial）
> 模拟竞争对手/敌对者视角，像憎恶项目一样尽可能找到缺陷。

| 维度 | 权重 | 9-10 | 7-8 | 5-6 | 3-4 | 0-2 |
|------|------|------|------|------|------|------|
| 技术脆弱性 | 20% | 架构无懈可击深度论证 | 架构合理无重大缺陷 | 基础实现有小问题 | 架构有明显缺陷 | 架构混乱/无文档 |
| 商业风险 | 20% | 市场数据详实护城河深 | 商业模式清晰有竞争力 | 基本可行风险可控 | 风险大于机会 | 商业逻辑不成立 |
| 执行能力 | 20% | 团队背景完美可执行 | 团队能力强经验匹配 | 基本能执行有小短板 | 执行力存疑 | 无团队背景信息 |
| 合规法律 | 20% | 合规设计完善无风险 | 主要合规项已覆盖 | 基础合规有风险点 | 合规风险较大 | 无任何合规设计 |
| 致命缺陷 | 20% | 无致命缺陷 | 有可控风险 | 存在潜在风险 | 有明确致命缺陷 | 多个致命缺陷 |

**通过阈值**: 加权总分 ≥ 5.0/10（敌对评审下通过难度更高）
**评分范围**: 基础分 35，封顶 75，保底 5
**敌对评审特点**: 每个缺陷自动推断关联缺陷（弱点放大），至少提出5个尖锐质疑问题

### 评审输出字段

每个评审返回：
| 字段 | 类型 | 说明 |
|------|------|------|
| `dimension_scores` | dict | 各维度 `{score: x/10, weight: y%, max: 10}` |
| `total_score` | float | 加权总分（0-10） |
| `pass_probability` | float | 通过率 = total_score / 10 |
| `rejection_reasons` | list | 硬性拒绝原因 |
| `suggestions` | list | 改进建议 |
| `sycophancy_warnings` | list | 抗讨好警告 |
| `depth_check` | object | 各层级检查覆盖率 |
| `hostile_questions` | list | 敌对评审的尖锐质疑（≥5个） |
| `weakness_chain` | list | 弱点放大链（原始缺陷→推断缺陷） |
| `kill_factors` | list | 致命缺陷列表 |

## 四、抗讨好机制（Anti-Sycophancy）

> 防止评审系统为了"讨好用户"而给出虚高分数。真实评审看到一堆缺陷时会越来越严厉，而非宽容。

### 4.1 缺陷计数与叠加惩罚

| 缺陷数 | 叠加惩罚 | 得分上限 | 说明 |
|--------|---------|---------|------|
| 0-1 | 无 | 无限制（可达 10 分） | 质量优秀，正常评分 |
| 2 | 无叠加 | 8.5/10 | 有小问题，不可超过 8.5 |
| 3 | -0.6 分 | 7.0/10 | 多项缺陷，叠加惩罚+上限 |
| ≥4 | -N×0.3 分 | 6.0/10 | 严重缺陷，评分封顶 6.0 |

### 4.2 非线性缺陷加权递减

当缺陷数 ≥ 3 时，pass_probability 按公式计算：
`final_score = base_score - Σ(defect_penalty_i × severity_i)`

其中 severity_i 随缺陷数递增：
- 第 1-2 个缺陷：severity = 1.0（线性惩罚）
- 第 3 个缺陷：severity = 1.3
- 第 4 个缺陷：severity = 1.7
- 第 5+ 个缺陷：severity = 2.0 + (n-5) × 0.5

**示例**: 5 个缺陷时，总惩罚 = p1×1.0 + p2×1.0 + p3×1.3 + p4×1.7 + p5×2.0，远高于简单累加。

### 4.3 高分但多缺陷的警告

当原始分 ≥ 8.0/10 但存在 ≥2 项缺陷时：
```
[抗讨好] 原始分 8.5/10 但存在 3 项缺陷，已校正至 7.0/10
```

### 4.4 硬性拒绝原因降档

当存在 `rejection_reasons` 时，即使原始分 ≥ 9.0/10：
```
[抗讨好] 高分(9.2/10)但有硬性拒绝原因，已降档
```

### 4.5 交叉验证规则

**新增**: 评审输出中的声明必须与输入信息交叉验证，防止"无证据高分"：

| 场景 | 触发条件 | 处理 |
|------|----------|------|
| 技术评审 | 声称"高可用"但无部署/容灾策略 | trust 扣 5 分，记录 sycophancy_warning |
| 投资评审 | 声称"百亿市场"但无数据来源/调研 | trust 扣 5 分 |
| 产品评审 | 声称"用户喜爱"但无验证数据/调研 | trust 扣 5 分 |
| 开源评审 | 声称"社区活跃"但无贡献者/issue 数据 | trust 扣 5 分 |
| 安全评审 | 安全评分≥7 但输入无安全设计描述 | 强制安全分降至 4，记录警告 |

### 4.6 基准对比规则

**新增**: 评审分数必须与行业基准对比，防止"闭门打分"：

| 基准维度 | 参考标准 | 偏差处理 |
|----------|----------|----------|
| 部署频率 | DORA 基准（Elite: 按需/High: 周/ Medium: 月/ Low: 年） | 声称"高频"但不符合基准→扣分 |
| 测试覆盖率 | 行业基线（80%+优秀/60%+良好/40%+及格） | 覆盖率<40%但给高分→强制降分 |
| 文档完整度 | CII Best Practices 要求 | 缺关键文档但给高分→强制降分 |
| 市场规模 | YC/a16z 投资基准 | 市场规模声明无数据→拒绝模式#6 |

### 4.7 核心原则

1. **分数就是分数** — 不搞 S/A/B/C/D 抽象等级，每个维度 0-10 分，总分 0-10 分
2. **缺陷越多扣越多** — 真实评审看到 5 个问题比看到 1 个问题严厉得多（非线性递减）
3. **天花板效应** — 多项缺陷时，即使其他维度完美，总分也有上限
4. **警告可见** — 用户能看到"原始分→校正分"的变化，了解系统没有虚报
5. **交叉验证** — 所有声明必须有输入信息中的证据支撑，无证据的高分自动降级

## 五、排查深度矩阵

> 每个评审类型必须覆盖以下 7 层检查，每层至少 1 个具体检查点。深度不足扣 3 分。

### 技术评审（7 层）
| 层级 | 检查项 | 通过标准 |
|------|--------|----------|
| 1. 架构 | 模块划分是否清晰？依赖方向是否单向？ | 有架构图+依赖说明 |
| 2. 代码 | 代码规范是否统一？有无测试？ | 有代码示例+覆盖率数据 |
| 3. 安全 | 输入校验/认证授权/数据加密是否覆盖？ | 至少覆盖 OWASP Top 3 |
| 4. 性能 | 有无性能基准？瓶颈分析？ | 有基准数据或预估 |
| 5. 依赖 | 依赖数量/版本/许可证是否明确？ | 依赖清单+许可证检查 |
| 6. 测试 | 测试策略（单元/集成/E2E）是否完整？ | 测试金字塔说明 |
| 7. 部署 | 部署流程/回滚方案/环境管理是否清晰？ | 部署脚本或 CI/CD 说明 |

### 投资评审（7 层）
| 层级 | 检查项 | 通过标准 |
|------|--------|----------|
| 1. 市场 | TAM/SAM/SOM 是否量化？增长率？ | 有数据来源+市场规模数值 |
| 2. 竞品 | 直接/间接竞品是否列出？差异化？ | 至少 2 个竞品对比 |
| 3. 团队 | 技术/产品/商业背景是否互补？ | 团队成员背景说明 |
| 4. 模式 | 营收路径是否清晰？定价策略？ | 有定价+营收模型 |
| 5. 风险 | 市场/技术/合规风险是否识别？ | 至少 3 类风险+缓解措施 |
| 6. 财务 | 预算/现金流/ROI 是否估算？ | 有财务预测+关键假设 |
| 7. 退出 | 退出策略是否明确？（收购/IPO/持续运营） | 有退出路径分析 |

### 产品评审（7 层）
| 层级 | 检查项 | 通过标准 |
|------|--------|----------|
| 1. 用户 | 用户画像是否具体？有场景？ | 至少 2 个用户画像+使用场景 |
| 2. 痛点 | 痛点是否真实？有验证？ | 用户调研/访谈/数据支撑 |
| 3. 方案 | 方案是否解决痛点？MVP 范围？ | 方案→痛点映射表 |
| 4. MVP | MVP 边界是否清晰？优先级？ | MoSCoW 优先级排序 |
| 5. 指标 | 成功指标是否量化？≥3 个？ | 有北极星指标+辅助指标 |
| 6. 反馈 | 用户反馈渠道和闭环是否建立？ | 有反馈机制说明 |
| 7. 迭代 | 迭代节奏和优先级管理是否明确？ | 有迭代计划和决策标准 |

### 开源评审（7 层）
| 层级 | 检查项 | 通过标准 |
|------|--------|----------|
| 1. 许可 | 许可证是否明确？兼容性问题？ | 有 LICENSE 文件+依赖许可证检查 |
| 2. 文档 | README/API/安装/示例是否齐全？ | 4 类文档均存在 |
| 3. 社区 | 贡献指引/行为准则/决策流程？ | CONTRIBUTING + CODE_OF_CONDUCT |
| 4. CI | 自动化测试/构建/发布流程？ | 有 CI/CD 配置 |
| 5. 依赖 | 第三方依赖声明？安全扫描？ | 有依赖清单+安全扫描结果 |
| 6. 治理 | 维护者列表/决策流程/发布节奏？ | 有治理文档 |
| 7. 安全 | 漏洞报告流程/安全策略？ | 有 SECURITY.md |

### 深度不足扣分规则
- 任一层级完全缺失 → 扣 3 分
- 任一层级检查项覆盖率 <50% → 扣 1.5 分
- 4 个评审类型中 ≥2 个深度不足 → 额外扣 5 分（系统性深度不足）

## 六、交叉验证规则

> 评审输出中的关键声明必须与输入信息交叉验证，防止无证据支撑的高分。

### 验证矩阵
| 声明类型 | 输入证据要求 | 缺失处理 |
|----------|-------------|----------|
| "高可用/高可靠" | 部署架构+容灾方案+SLA声明 | trust-5，记录"声明无证据" |
| "大规模市场" | 数据来源+TAM计算过程 | trust-5，触发拒绝模式#6 |
| "用户验证/喜爱" | 调研数据/用户反馈/MVP测试 | trust-5，触发拒绝模式#14 |
| "社区活跃/成熟" | 贡献者数/PR合并/Issue响应 | trust-5，触发开源评审降级 |
| "安全性好" | 安全设计描述/审计结果/OWASP覆盖 | 安全分降至4，记录警告 |
| "性能优秀" | 基准测试数据/负载测试结果 | 性能分上限6，记录"无基准" |
| "成本低/高效" | 成本计算/效率对比数据 | outcome降级为"未量化" |

### 验证执行流程
1. 提取评审输出中的所有关键声明（含形容词+量化描述）
2. 在输入信息中搜索对应证据
3. 无证据支撑的声明 → 降级对应维度分数 + 记录 warning
4. 最终输出中标注"已交叉验证：N 条声明 / M 条有证据支撑"
