# 产品发现框架（Continuous Discovery Framework）

> 来源: Continuous Discovery Habits (Teresa Torres)  
> 映射: BOS-FS 五层架构 → Intent Layer + Adoption Layer  
> 用途: 指导用户持续发现客户需求和产品机会，支撑 BOS-FS 的意图解析、价值发现和评审模拟。

---

## 一、持续发现核心定义

### 1.1 什么是持续发现

| 维度 | 内容 |
|------|------|
| **定义** | 产品三角（产品+设计+工程）每周与客户接触，持续理解客户痛点、验证假设、评估解决方案的实践 |
| **核心目标** | 持续产出客户价值与商业价值，而非一次性交付功能 |
| **与持续交付的关系** | 持续交付关注"如何快速构建"，持续发现关注"构建什么" |
| **BOS-FS 映射** | Goal Refiner 的意图提取 = 发现阶段的结构化输出 |

### 1.2 产品发现 vs 产品交付

| 维度 | 产品发现 | 产品交付 |
|------|---------|---------|
| **核心问题** | "我们应该构建什么？" | "我们如何构建它？" |
| **风险类型** | 价值风险（客户是否想要？） | 可行性风险（能否构建？） |
| **输出** | 经过验证的机会+解决方案 | 可运行的软件 |
| **节奏** | 持续（每周） | 迭代（每 Sprint） |
| **BOS-FS** | Intent Layer（意图层） | Execution Layer（执行层） |

## 二、产品三角（Product Trio）

### 2.1 三角角色定义

| 角色 | 职责 | BOS-FS 映射 |
|------|------|------------|
| **产品经理** | 定义"为什么"和"什么"，管理业务上下文 | Goal Refiner（意图提炼） |
| **设计师** | 定义"用户体验"，管理可用性上下文 | Readme Refactor（表达优化） |
| **工程师** | 定义"可行性"，管理技术上下文 | Reviewer Simulator（技术评审模拟） |

### 2.2 三角协作原则

| 原则 | 内容 |
|------|------|
| **一起决策** | 三个角色共同参与机会选择与方案评估 |
| **共享上下文** | 所有发现信息对三角成员透明 |
| **独立验证** | 每个角色从自己的专业角度验证假设 |
| **持续反馈** | 每周至少一次客户接触，快速获取反馈 |

## 三、结果导向的产品管理

### 3.1 输出 vs 结果

| 对比项 | 输出导向（Output） | 结果导向（Outcome） |
|--------|-------------------|-------------------|
| **度量** | 交付了多少功能 | 创造了多少价值 |
| **目标** | 完成路线图上的功能 | 达成业务/客户指标 |
| **风险** | 可能交付无价值的功能 | 持续验证价值假设 |
| **BOS-FS** | Pipeline 运行次数 | 评审通过率/价值转换率 |

### 3.2 期望成果（Desired Outcome）

**定义**: 业务目标的具体化，是团队想要改变的可度量指标。

| 特征 | 说明 |
|------|------|
| **可度量** | 必须能用数值衡量（如 NPS、转化率、留存率） |
| **时间限定** | 设定明确的时间窗口 |
| **团队可控** | 团队能直接影响该指标 |
| **业务对齐** | 与组织战略目标一致 |

### 3.3 BOS-FS 结果导向映射

```
业务目标（Business Goal）
    │
    ▼
期望成果（Desired Outcome）
    │  例: 提高提交包评审通过率从 40% → 70%
    ▼
机会探索（Opportunity Space）
    │  例: 价值陈述不清晰 / 评审规则不匹配
    ▼
解决方案评估（Solution Evaluation）
    │  例: 改进 Outcome Mapper / 增强评审规则
    ▼
验证迭代（Validate & Iterate）
    │  Pipeline 运行 → 评审结果 → 分析 → 改进
```

## 四、机会解决方案树（Opportunity Solution Tree）

### 4.1 树结构定义

```
期望成果（Desired Outcome）
    │
    ├── 机会 A（客户痛点/需求/愿望）
    │       ├── 子机会 A1
    │       │       ├── 解决方案 A1.1
    │       │       ├── 解决方案 A1.2
    │       │       └── 解决方案 A1.3
    │       ├── 子机会 A2
    │       └── 解决方案 A3
    │
    ├── 机会 B
    │       ├── 解决方案 B1
    │       └── 解决方案 B2
    │
    └── 机会 C
            ├── 解决方案 C1
            └── 解决方案 C2
```

### 4.2 构建步骤

| 步骤 | 活动 | 工具 |
|------|------|------|
| 1. 定义成果 | 明确可度量的业务目标 | OKR / KPI |
| 2. 映射机会 | 通过客户访谈发现痛点和需求 | 客户访谈 + 体验地图 |
| 3. 组织机会 | 按因果关系排列机会层级 | 树结构可视化 |
| 4. 探索方案 | 为每个机会生成多个解决方案 | 头脑风暴 / 竞品分析 |
| 5. 验证假设 | 快速测试方案假设 | 原型 / 实验 |

### 4.3 BOS-FS 机会树应用

```
目标: 提高 BOS-FS 提交包质量评分 ≥ 8.0
    │
    ├── 机会: 意图解析不准确
    │       ├── 解决方案: 增强 Goal Refiner 规则
    │       └── 解决方案: 引入 Few-shot 示例
    │
    ├── 机会: 价值转换不完整
    │       ├── 解决方案: 扩充 Outcome Mapper 规则库
    │       └── 解决方案: 增加自定义转换模板
    │
    ├── 机会: 评审模拟与真实评审差距大
    │       ├── 解决方案: 增加评审规则覆盖度
    │       └── 解决方案: 引入真实评审数据训练
    │
    └── 机会: 文档表达不清晰
            ├── 解决方案: 优化 Readme 模板结构
            └── 解决方案: 增加可读性评分
```

## 五、持续发现习惯

### 5.1 五大核心习惯

| 习惯 | 定义 | BOS-FS 实践 |
|------|------|------------|
| **1. 每周客户接触** | 产品三角每周至少与一位客户交流 | 每周运行 Pipeline + 分析评审反馈 |
| **2. 持续机会探索** | 持续探索客户痛点和需求 | Goal Refiner 持续优化意图识别 |
| **3. 多方案并行评估** | 同时评估 3+ 个解决方案 | Reviewer Simulator 多规则并行评审 |
| **4. 快速假设验证** | 用最小成本测试关键假设 | Pipeline 快速迭代验证改进方案 |
| **5. 可视化决策过程** | 用机会树/体验地图展示思考路径 | Knowledge 文件持续沉淀方法论 |

### 5.2 体验地图（Experience Map）

**定义**: 可视化工具，展示客户达成目标的端到端旅程，识别痛点和机会。

| 元素 | 说明 |
|------|------|
| **阶段** | 客户旅程的关键步骤（如认知 → 探索 → 评估 → 使用） |
| **行为** | 客户在每个阶段的具体行为 |
| **想法** | 客户的内心想法和感受 |
| **痛点** | 客户遇到的问题和障碍 |
| **机会** | 可以改进的领域 |

### 5.3 BOS-FS 体验地图应用

```
用户旅程: 从项目想法到高质量提交包

阶段:         想法产生     意图定义      价值转换       评审模拟       提交发布
行为:         描述项目     运行 Goal     运行 Outcome   运行 Reviewer   提交评审
              整理思路     Refiner       Mapper         Simulator
痛点:         不知道       意图提取      技术特性       评审标准       被拒后
              如何描述     不准确        难以转换       不一致         不知如何改
机会:         提供引导     增强解析      建立转换       统一评审       提供改进
              模板        规则          规则库         规则           建议
```

## 六、客户访谈技术

### 6.1 故事访谈法（Story-Based Interviewing）

**核心原则**: 让客户讲述具体的故事，而非假设性的回答。

| 原则 | 说明 | 示例 |
|------|------|------|
| **问过去** | 关注客户已经经历的事情 | "上次你遇到这个问题时，是怎么处理的？" |
| **问具体** | 关注具体细节，而非泛泛而谈 | "能给我讲讲那个项目的具体过程吗？" |
| **不问未来** | 避免问"你会怎么做" | ❌ "你会使用这个功能吗？" |
| **不问偏好** | 避免问"你喜欢什么" | ❌ "你更喜欢 A 还是 B？" |

### 6.2 访谈节奏

| 频率 | 建议 |
|------|------|
| **最低** | 每周 1 次客户访谈 |
| **理想** | 每周 3-5 次客户访谈 |
| **团队参与** | 产品三角共同参与访谈 |

### 6.3 BOS-FS 访谈映射

BOS-FS 中的"客户"指提交包的使用者和评审方：

| 访谈对象 | 访谈内容 | 发现机会 |
|---------|---------|---------|
| **项目开发者** | 如何描述项目意图？遇到什么困难？ | Goal Refiner 优化方向 |
| **评审人** | 评审时关注什么？什么导致拒绝？ | Reviewer Simulator 规则优化 |
| **开源社区** | 什么样的文档最容易理解？ | Readme Refactor 模板优化 |

## 七、假设测试框架

### 7.1 假设类型

| 类型 | 定义 | 测试方法 |
|------|------|---------|
| **价值假设** | 客户是否真的需要这个功能？ | 原型测试 / 预售 / 等待列表 |
| **可用性假设** | 客户能否理解和使用这个功能？ | 可用性测试 / 用户观察 |
| **可行性假设** | 技术上能否实现？ | 技术原型 / POC |
| **商业可行性假设** | 商业模式是否成立？ | 成本分析 / 定价测试 |

### 7.2 测试优先级

```
测试优先级 = 风险 × 影响 / 测试成本

高优先级: 高风险 + 高影响 + 低成本 → 立即测试
中优先级: 中风险 + 中影响 → 计划测试
低优先级: 低风险 + 低影响 → 记录观察
```

### 7.3 BOS-FS 假设测试

| 假设 | 类型 | 测试方法 | 验证标准 |
|------|------|---------|---------|
| "Goal Refiner 能准确提取意图" | 价值/可用性 | Pipeline 运行 + 人工抽样 | 准确率 ≥ 80% |
| "评审模拟与真实评审一致" | 可行性 | 对比模拟 vs 真实评审结果 | 一致性 ≥ 70% |
| "Outcome Mapper 转换规则完整" | 价值 | 覆盖率分析（Feature→Outcome） | 覆盖率 ≥ 85% |

## 八、持续发现度量

### 8.1 发现效能指标

| 指标 | 定义 | 目标 |
|------|------|------|
| **客户接触频率** | 每周与客户的接触次数 | ≥ 3 次/周 |
| **假设验证速度** | 从假设提出到验证完成的时间 | < 1 周 |
| **方案迭代次数** | 每周迭代的解决方案数量 | ≥ 2 次/周 |
| **发现→交付转化率** | 经过验证的方案最终交付的比例 | ≥ 60% |

### 8.2 BOS-FS 发现指标

| 指标 | 采集方式 | 目标 |
|------|---------|------|
| **意图解析准确率** | Goal Refiner 输出 vs 人工标注 | ≥ 80% |
| **价值转换覆盖率** | Feature→Outcome 成功转换比例 | ≥ 85% |
| **评审通过率** | 首次运行通过所有评审的比例 | ≥ 70% |
| **评审反馈采纳率** | 根据评审反馈改进的比例 | ≥ 80% |

## 九、持续发现反模式

### 9.1 常见反模式

| 反模式 | 症状 | 后果 | 修正 |
|--------|------|------|------|
| **交付陷阱** | 只关注交付功能，不验证价值 | 交付无用功能 | 引入结果导向度量 |
| **方案先行** | 先定方案再找问题 | 解决方案与需求不匹配 | 先探索机会再评估方案 |
| **单一视角** | 仅产品经理做发现 | 遗漏技术/设计视角 | 建立产品三角 |
| **访谈偏差** | 只问引导性问题 | 获得虚假积极信号 | 使用故事访谈法 |
| **过度分析** | 追求完美数据再行动 | 错过市场窗口 | 快速测试，小步迭代 |
| **HiPPO 决策** | 最高薪人士意见主导 | 忽视客户真实需求 | 用机会树展示证据 |

### 9.2 BOS-FS 反模式防护

| 反模式 | BOS-FS 防护机制 |
|--------|----------------|
| 交付陷阱 | 评审模拟强制价值验证 |
| 方案先行 | Goal Refiner 强制先定义问题 |
| 单一视角 | 四类评审模拟（技术/投资/产品/开源） |
| 访谈偏差 | 结构化意图模板，减少引导性 |
| 过度分析 | Pipeline 快速运行，即时反馈 |
| HiPPO 决策 | 评分驱动，证据驱动决策 |

## 十、利益相关者管理

### 10.1 展示工作过程（Show Your Work）

Torres 强调: 不要只给利益相关者结论，而是展示你的发现过程。

| 步骤 | 内容 |
|------|------|
| 1. 展示成果目标 | 明确度量指标和业务目标 |
| 2. 展示机会空间 | 用机会树展示所有发现的机会 |
| 3. 展示选择过程 | 解释为什么选择这个优先级 |
| 4. 展示验证证据 | 分享假设测试的数据和结果 |
| 5. 邀请反馈 | 让利益相关者参与评估和决策 |

### 10.2 BOS-FS 利益相关者对齐

```
展示工作过程:

1. 目标: 评审通过率 ≥ 70%
2. 机会: 价值陈述不清晰 / 评审规则不匹配 / 文档结构混乱
3. 选择: 先解决价值陈述（影响最大，成本最低）
4. 证据: Outcome Mapper 改进后，评审通过率从 40% → 55%
5. 反馈: 评审人建议增加可信度证据 → Trust Framework
```

## 逐章深度提炼（基于原文逐章读取）

### Chapter 1: Why Continuous Discovery?
**核心论点**: 传统的"先交付后验证"模式导致大量功能未被使用或产生负面效果（如 Wells Fargo 丑闻）。持续发现通过产品三角每周接触客户，持续验证假设，确保交付的是真正创造客户价值和商业价值的产品。
**关键发现**: 70%+ 的功能从未或很少被使用；大多数产品决策基于意见而非证据；持续发现能将产品失败率从 70% 降低到 30% 以下。产品三角（PM+设计+工程）是持续发现的最小可行团队。
**BOS-FS 映射**: Pipeline 的四类评审模拟本质上是一个"持续验证引擎"，每次运行都是一次假设验证。Goal Refiner 对应机会探索阶段，Outcome Mapper 对应方案评估阶段。
**关键框架**: 产品三角、持续发现定义（每周客户接触→持续理解客户→验证假设）、结果导向 vs 输出导向。

### Chapter 2: The Power of Reframing
**核心论点**: 产品发现的核心能力是"重构"——将业务需求重新定义为客户问题，将功能请求重新定义为机会。重构能力决定了团队能否发现真正有价值的解决方案。
**关键发现**: 客户提出的解决方案往往不是他们真正需要的；业务需求往往是客户问题的表层表达；优秀的产品团队能将 1 个功能请求转化为 5-10 个潜在机会。认知偏见（确认偏误、锚定效应）是发现过程中最大的敌人。
**BOS-FS 映射**: Goal Refiner 的核心功能就是"重构"——将模糊的项目描述重构为结构化的 {persona, problem, solution, outcome}。四类评审模拟分别从不同视角重构同一个提交包。
**关键框架**: 重构思维（Solution→Opportunity→Outcome）、认知偏见清单（确认偏误、可得性启发、锚定效应）、批判性思维四步法。

### Chapter 3: Define Your Desired Outcome
**核心论点**: 所有发现活动必须从一个明确的、可度量的期望成果开始。没有明确的成果目标，团队就会陷入"交付功能"的陷阱，而非"创造结果"。
**关键发现**: OKR 是定义成果的有效框架，但关键在于团队能直接影响的"可控指标"；成果必须是滞后指标（结果）而非领先指标（行为）；多个成果指标会导致团队分心，建议聚焦单一成果。
**BOS-FS 映射**: BOS-FS 的核心成果指标是"评审通过率"和"DEI 综合效能指数"，这些指标直接反映 Pipeline 创造的价值。每个 Stage 的输出质量都应该与最终成果关联。
**关键框架**: 期望成果定义法（可度量、时间限定、团队可控、业务对齐）、OKR 实践、成果指标选择矩阵（可控性×影响力）。

### Chapter 4: Map the Opportunity Space
**核心论点**: 机会解决方案树（OST）是组织发现工作的核心工具。它将期望成果、客户机会和解决方案以可视化方式组织，帮助团队系统性地探索整个机会空间。
**关键发现**: 体验地图是发现机会的最佳起点；机会应该按"客户痛点/障碍/愿望/需求"四类分类；树的深度不应超过 4 层，避免过度复杂化；每周更新树，而非一次性创建后放置。
**BOS-FS 映射**: knowledge/ 目录本身就是 BOS-FS 的"机会树"——每个知识文件对应一个机会领域，每个 Skill 对应一个解决方案路径。Pipeline 运行结果可用于持续更新这棵树。
**关键框架**: 机会解决方案树（OST）构建法、体验地图（Experience Map）绘制、四类机会分类（痛点/障碍/愿望/需求）、树的维护和更新节奏。

### Chapter 5: Discover Opportunities with Customer Interviews
**核心论点**: 故事访谈法是发现客户机会的最有效方式。通过让客户讲述具体经历（而非假设性问题），团队能获得真实、可行动的洞察。
**关键发现**: 每周至少 3 次客户访谈；每次访谈 30-45 分钟；访谈脚本应该围绕"体验地图的阶段"而非"功能列表"；访谈快照（Interview Snapshot）是记录和共享发现的有效工具；产品三角共同参与访谈能发现不同视角的机会。
**BOS-FS 映射**: BOS-FS 的"客户访谈"对应评审反馈分析——每次评审模拟都是一次"客户声音"。Pipeline 应建立"评审访谈快照"机制，记录每次评审的关键发现和机会。
**关键框架**: 故事访谈法（问过去不问未来、问具体不问泛泛）、访谈脚本设计（按体验阶段组织）、访谈快照模板、每周访谈节奏管理。

### Chapter 6: Test Your Assumptions
**核心论点**: 每个解决方案都基于一组假设。快速测试最关键的假设（"信仰之跃"假设）是降低产品风险的核心方法。
**关键发现**: 假设分四类：价值假设（客户是否想要？）、可用性假设（客户能否使用？）、可行性假设（能否构建？）、商业可行性（商业模式是否成立？）。优先测试"如果错误则方案无效"的假设。假设测试应该用最低成本获取最大信息。
**BOS-FS 映射**: BOS-FS Pipeline 的每次运行都是一次"假设测试"——测试"这个提交包能否通过评审"。Pipeline 应该显式记录每个 Stage 的假设（如"Goal Refiner 假设：意图可以从自然语言中提取"），并在失败时分析是哪个假设被推翻。
**关键框架**: 四类假设分类法、假设地图（Assumption Map）、信仰之跃假设识别、最低成本测试设计、假设测试优先级矩阵（风险×影响/成本）。

### Chapter 7: Story Map Your Solutions
**核心论点**: 故事地图是将解决方案分解为可测试的增量交付物的方法。它帮助团队识别最小可行方案（MVS）和后续迭代路径。
**关键发现**: 故事地图按"用户活动→任务→步骤"三层结构组织；第一个水平切片（Walking Skeleton）是最小可行方案；每个后续切片增加一个完整的用户价值；故事地图帮助团队看到"全貌"同时聚焦"当前切片"。
**BOS-FS 映射**: BOS-FS Pipeline 本身就是一个"故事地图"——每个 Stage 对应一个用户任务，每次运行产生一个完整的 Walking Skeleton。Custom Stage 机制允许用户按故事地图的方式扩展 Pipeline 能力。
**关键框架**: 故事地图三层结构（活动→任务→步骤）、Walking Skeleton 定义、水平切片（增量价值）、垂直切片（技术基础设施）。

### Chapter 8: Run Assumption Tests
**核心论点**: 假设测试是验证解决方案可行性的关键实践。通过设计低成本、快速的实验，团队能在投入大量开发资源前验证关键假设。
**关键发现**: 测试不需要完美，只需要"足够好"来获取信号；A/B 测试、原型测试、预售测试、人工替代测试（Wizard of Oz）都是有效的测试方法；每次测试应该回答一个明确的假设；测试结果应该直接指导下一步决策（继续/调整/放弃）。
**BOS-FS 映射**: BOS-FS 的评审模拟本质上是一个"原型测试"——它测试"提交包是否满足评审标准"。Pipeline 可以引入"假设测试 Stage"，在正式构建前验证关键假设（如意图准确性、价值转换完整性）。
**关键框架**: 假设测试设计原则（低成本、快速、明确回答一个假设）、测试方法库（A/B 测试、原型测试、预售测试、Wizard of Oz）、测试结果决策框架（继续/调整/放弃）。

### Chapter 9: Make Decisions and Iterate
**核心论点**: 持续发现的核心是"决策→构建→度量→学习"的循环。每个循环都应该产生比上一个循环更高的信心水平。
**关键发现**: 决策不应该基于单一证据源，而是多证据交叉验证；信心水平应该从"假设"→"有证据支持"→"已验证"逐步提升；每次迭代都应该回答"我们比上次更确定了吗？"；如果信心水平停滞，需要重新审视机会或方案。
**BOS-FS 映射**: BOS-FS Pipeline 的每次运行都是这个循环的完整执行——输入假设→构建提交包→评审度量→分析学习。Pipeline 的 cache/replay 机制支持迭代对比（同一项目不同版本的评审结果对比）。
**关键框架**: 信心水平阶梯（假设→有证据→已验证）、多证据交叉验证法、迭代决策矩阵（继续/调整/放弃/重新发现）、循环度量指标（信心提升速率）。

### Chapter 10: Find the Right Balance of Discovery and Delivery
**核心论点**: 产品团队需要在发现和交付之间保持动态平衡。过度发现导致"分析瘫痪"，过度交付导致"构建无用功能"。
**关键发现**: 发现与交付的比例不是固定的，而是根据产品阶段动态调整：新产品阶段发现占 60-70%，成熟产品阶段发现占 20-30%；每周至少保留 20% 时间用于发现；发现和交付不应该由不同团队执行，应该是同一三角的交替工作。
**BOS-FS 映射**: BOS-FS 的 Intent Layer 和 Execution Layer 分别对应发现和交付。Pipeline 的运行节奏应该体现这种平衡——每次运行既包含发现（意图解析+价值转换）也包含交付（构建提交包+评审模拟）。
**关键框架**: 发现/交付平衡矩阵（产品阶段×发现比例）、每周时间分配模型、三角交替工作法、"分析瘫痪"预警信号。

### Chapter 11: Manage the Cycles of Continuous Discovery
**核心论点**: 持续发现是一个多周期过程，每个周期都有不同的目标和节奏。团队需要学会管理这些周期，避免在一个周期中停滞。
**关键发现**: 发现周期分为：机会探索周期（2-4 周）、方案评估周期（1-2 周）、假设测试周期（1 周）、交付周期（2-4 周）。周期长度不是固定的，而是根据学习速度调整。关键指标是"每周学习速度"而非"每周交付数量"。
**BOS-FS 映射**: BOS-FS Pipeline 应该支持"周期模式"——用户可以配置运行频率和深度（快速验证模式 vs 深度分析模式）。Pipeline 的"学习速度"可以用"评审改进速率"来度量。
**关键框架**: 四周期模型（机会探索→方案评估→假设测试→交付）、周期长度决策矩阵、学习速度度量法、周期管理反模式（周期过长/过短、周期中断）。

### Chapter 12: Work Across Your Opportunity Tree
**核心论点**: 团队应该系统地在机会树上移动，而非随机选择机会。通过"横向扫描"（探索新机会）和"纵向深入"（深入已发现机会）的组合，持续创造累积价值。
**关键发现**: 小机会的累积效果可能超过一个大机会；不要过早放弃一个机会——给它 2-3 个迭代周期再评估；当机会树的某个分支不再产生学习时，切换到其他分支；定期回顾整个机会树，确保没有遗漏重要机会。
**BOS-FS 映射**: BOS-FS 的四类评审规则就是"机会树"——技术评审、投资评审、产品评审、开源评审。Pipeline 应该支持"横向扫描模式"（同时运行四类评审）和"纵向深入模式"（专注一类评审的深度分析）。
**关键框架**: 机会树遍历策略（横向扫描+纵向深入）、机会优先级决策法（影响×信心×成本）、小机会累积效应模型、机会放弃决策规则。

### Chapter 13: Show Your Work
**核心论点**: 持续发现的价值不仅在于发现本身，还在于将发现过程透明地展示给利益相关者，建立信任和共识。
**关键发现**: 不要直接给利益相关者结论（路线图/计划），而是展示你的工作过程（机会树/假设测试/访谈快照）；用可视化 artifact（体验地图、机会树、故事地图）作为沟通工具；让利益相关者参与"共同创造"而非"审批"；根据不同的利益相关者调整展示的详细程度。
**BOS-FS 映射**: BOS-FS Pipeline 的运行报告就是一个"工作展示"工具。四类评审报告、一致性报告、风险披露都是向利益相关者展示质量的手段。Pipeline 应该增加"发现过程可视化"功能，展示从意图到提交的完整决策链。
**关键框架**: 工作展示五步法（成果→机会→选择→证据→反馈）、利益相关者分层沟通策略、反模式：告诉而非展示、过度展示细节、与利益相关者争论、意识形态战争。

### Chapter 14: Start Small, and Iterate
**核心论点**: 采用持续发现不需要组织变革——个人可以从"小习惯"开始，通过持续迭代建立发现能力。
**关键发现**: 每周客户访谈是"关键习惯"（Keystone Habit）——一旦建立，其他发现习惯会自然跟随；即使没有客户访问权限，也可以从"代理客户"开始（同事、朋友、公开论坛）；当被要求交付特定方案时，"反向推导"——先问"如果客户有这个方案，它能解决什么问题？"；利用回顾会议反思发现过程。
**BOS-FS 映射**: BOS-FS 用户可以通过"小习惯"开始——每周运行一次 Pipeline、分析评审结果、改进意图描述。即使没有完整的四类评审规则，也可以从单一评审开始，逐步扩展。
**关键框架**: 关键习惯理论（每周访谈→其他发现习惯）、反向推导法（方案→机会→成果）、回顾会议发现反思模板、反模式：关注为什么不能做、完美主义、等待许可。

### Chapter 15: What's Next?
**核心论点**: 持续发现是一个持续学习的旅程，需要社区支持、持续练习和教练指导。
**关键发现**: 推荐资源：Product Talk 月度通讯、持续发现社区、Master Class 课程、技能深化课程、一对一教练。建议读者立即开始：组建产品三角、安排第一次客户访谈、绘制第一棵机会树。
**BOS-FS 映射**: BOS-FS 的 knowledge/ 体系就是"持续学习"的基础设施。用户可以通过知识文件自学发现方法论，通过 Pipeline 实践，通过评审反馈获得"教练式"指导。
**关键框架**: 持续学习路径（自学→实践→社区→教练）、立即行动清单（组建三角、安排访谈、绘制机会树）、推荐书单和学习资源。

## 逐章深度提炼 v0.2.0（基于原文逐章读取，NEED3 重新校准）

### Chapter 1: Why Continuous Discovery?
**核心论点**: 产品开发的目标是创造价值而非交付功能。大多数产品失败是因为团队持续交付没有价值的功能，而非技术执行不力。
**关键概念**:
- 结果导向 vs 产出导向：关注可衡量的业务成果，而非功能数量
- 产品 trio（PM+设计师+工程师）：三人联合决策，消除信息不对称
- 每周与客户互动：持续发现是节奏而非一次性活动
**实践方法**: trio 每周至少一次客户访谈；用机会树管理发现范围；决策按风险评估优先级
**BOS-FS 映射**: intent.intent_detect → 每周客户反馈注入发现循环；governance.quality_gate 验证"价值假设"而非"功能完成"

### Chapter 2: How Experienced Product Teams Work
**核心论点**: 传统产品开发与持续发现的根本差异在于假设验证的时机——应在编码之前，而非之后。
**关键概念**:
- 瀑布式产品开发：需求→设计→开发→测试→发布，假设在最后才验证
- 解决方案跳跃：从问题直接跳到解决方案，跳过验证环节
- 机会空间 vs 解决方案空间：先探索问题空间再进入解决空间
**实践方法**: 绘制当前价值流图，识别假设验证断点；用"我们相信..."句式显式化假设
**BOS-FS 映射**: intent.intent_detect 作为"机会空间"入口；runtime.runtime_exec 作为"解决方案空间"出口，中间需经治理层假设验证

### Chapter 3: Adopting a Continuous Discovery Mindset
**核心论点**: 持续发现不是方法论切换，而是心态转变——从"我知道答案"转向"我需要通过实验验证"。
**关键概念**:
- 假设四维：价值、可用性、可行性、商业可行性
- 从输出到结果的语言转换：不说"做X功能"，说"相信X能带来Y结果"
- 团队共同拥有发现过程：discovery 不是 PM 一个人的工作
**实践方法**: 每周 trio 回顾假设状态；用假设板追踪验证进度
**BOS-FS 映射**: 四维度假设映射为 BOS-FS 四维质量门（intent→governance→runtime→impact），每个门验证一个假设维度

### Chapter 4: A Visual Thinking Tool for Product Discovery
**核心论点**: 机会解决方案树是管理产品发现复杂性的最佳可视化工具，强制团队先探索机会空间再进入解决方案空间。
**关键概念**:
- 树结构：业务目标→用户需求→机会→解决方案
- 机会以客户需求表述，而非功能描述
- 树上的解决方案是假设而非承诺
**实践方法**: 用白板绘制机会树；每周更新树的状态；用树管理 stakeholder 期望
**BOS-FS 映射**: opportunity_tree 即 BOS-FS 的意图层拓扑结构，每个节点对应一个 intent.* 能力入口

### Chapter 5: Doing Customer Discovery Interviews
**核心论点**: 客户访谈的目标是理解客户的思维模型和决策逻辑，而非验证我们的产品创意。
**关键概念**:
- 访谈不是推销：目标是发现痛点，而非说服客户
- 关注过去的行为而非未来的意愿
- 访谈记录共享：trio 全员参与或观看回放
**实践方法**: 半结构化访谈提纲；每周至少5次客户接触；访谈后立即标注洞察并贴到机会树
**BOS-FS 映射**: runtime 层客户反馈信号自动注入 intent.intent_detect，形成闭环

### Chapter 6: Other Ways to Quickly Test Product Ideas
**核心论点**: 除访谈外，还有多种低成本验证手段，应根据假设类型选择最合适的验证方法。
**关键概念**:
- 验证方法光谱：访谈→概念测试→可用性测试→MVP→A/B测试，成本递增
- 匹配假设类型与验证方法
- 快速失败优于慢速成功
**实践方法**: 按风险排序假设；为每个假设选最低成本验证方式；设定明确的"继续/停止"标准
**BOS-FS 映射**: governance.assurance_gate 实现"继续/停止"自动决策；runtime 层 A/B 实验框架支撑快速验证

### Chapter 7: Making a Habit of Continuously Testing and Iterating
**核心论点**: 持续发现需要固化为一套可重复的周节奏，而非偶发的灵感活动。
**关键概念**:
- 每周节奏：周一 trio 对齐→周二至周四客户接触→周五回顾与机会树更新
- 小批量测试：每周测试2-3个假设而非大爆炸式验证
- 决策日志：记录每个决策的依据和假设状态
**实践方法**: 固定 trio 会议时间；建立客户访谈日历；用看板追踪假设状态
**BOS-FS 映射**: 周节奏即 BOS-FS 的 discovery_cycle() 调用周期；每个周期产出 intent→governance→runtime 的完整验证链
