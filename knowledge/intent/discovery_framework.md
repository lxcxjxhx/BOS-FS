# 发现框架（Discovery Framework）

> 来源: Continuous Discovery Habits (Teresa Torres) / Product Management in Practice (Matt LeMay)  
> 映射: BOS-FS 五层架构 → Intent Layer (`01_intent_parser/goal_refiner.py`)  
> 用途: 指导用户在提交前完成结构化发现，确保项目意图基于真实需求而非假设。

---

## 一、核心发现原则

### 1.1 Outcome over Output（结果优于产出）

| 维度 | 内容 |
|------|------|
| **核心命题** | 不要度量"做了什么"，要度量"改变了什么" |
| **常见误区** | 以功能数量、代码行数、提交次数作为成功指标 |
| **正确做法** | 以用户行为变化、业务指标改善、效率提升作为成功指标 |
| **BOS-FS 映射** | Goal Refiner 强制输出 `outcome` 字段，拒绝无目标的纯功能描述 |

**实战检查**:
- ❌ "我们做了用户管理系统"
- ✅ "用户管理系统上线后，注册转化率从 12% 提升到 28%"

### 1.2 持续发现习惯

Teresa Torres 定义的五项核心习惯，直接映射到 BOS-FS 工作流:

| 习惯 | Torres 原文要点 | BOS-FS 中的对应 |
|------|-----------------|-----------------|
| **拆解大目标** | 将愿景拆解为可验证的结果 | `Goal Refiner` 将模糊意图拆解为 `{persona, problem, solution, outcome}` |
| **绘制机会树** | 以目标为根，以用户/痛点/机会为分支 | 见下方「机会树模板」 |
| **持续用户访谈** | 每周至少与一位用户对话 | 提交前需回答「用户验证清单」(见 §六) |
| **假设驱动开发** | 所有功能都是待验证假设 | `Outcome Mapper` 将 Feature 标注为待验证假设 |
| **每周回顾** | 定期审视发现结果与方向 | Pipeline 每次运行生成结构化报告供回顾 |

### 1.3 双轨发现模型（Dual-Track Discovery）

```
┌──────────────────────┐     ┌──────────────────────┐
│   Discovery Track    │────▶│  Delivery Track      │
│   (发现轨道)         │     │  (交付轨道)          │
│                      │     │                      │
│ • 问题空间探索       │     │ • 解决方案实现       │
│ • 用户访谈/验证      │     │ • 代码/文档/提交包   │
│ • 机会识别与优先级   │     │ • 测试与质量保障     │
│ • 假设验证           │     │ • 评审模拟           │
└──────────────────────┘     └──────────────────────┘
         │                           │
         └────── 每周同步 ──────────┘
```

**BOS-FS 中的应用**:
- Discovery Track → `Goal Refiner` + `Outcome Mapper` (Intent Layer)
- Delivery Track → `Readme Refactor` + `Submission Builder` + `Reviewer Simulator` (Execution/Runtime Layer)
- 两轨之间的同步 → Pipeline 的 `PipelineContext.shared_data` 数据流

## 二、机会树框架（Opportunity Solution Tree）

### 2.1 树结构定义

```
期望结果 (Outcome)
├── 用户群 A
│   ├── 痛点 A1
│   │   ├── 机会 A1-1 (如何让用户…？)
│   │   └── 机会 A1-2
│   └── 痛点 A2
│       └── 机会 A2-1
├── 用户群 B
│   └── 痛点 B1
│       └── 机会 B1-1
└── 用户群 C
    └── 痛点 C1
        └── 机会 C1-1
```

### 2.2 BOS-FS 机会树构建步骤

**步骤 1 — 明确期望结果** (对应 `Goal Refiner` 的 `outcome` 字段):
- 从项目描述中提取量化目标
- 如果未量化，标注"未明确"并要求补充
- 示例: "将用户注册转化率从 12% 提升到 28%"

**步骤 2 — 识别用户群体** (对应 `persona` 字段):
- 谁受到这个问题的影响？
- 不同用户群可能有不同痛点
- 示例: 新用户 / 老用户 / 管理员

**步骤 3 — 映射痛点** (对应 `problem` 字段):
- 每个用户群面临的核心障碍是什么？
- 使用"5 Whys"追问根因
- 示例 (新用户): "注册流程太长 (12步) → 用户中途流失"

**步骤 4 — 发现机会** (对应 `solution` 字段):
- 针对每个痛点，问"如何让用户…？"
- 生成 2-3 个机会点
- 示例: "如何将注册步骤从 12 步减少到 3 步？"

**步骤 5 — 优先级排序**:
- 以影响力 × 可行性评分排序
- 优先验证高风险假设

### 2.3 机会树与 BOS-FS Pipeline 映射

| 机会树节点 | BOS-FS 字段 | Stage | 知识文件 |
|-----------|-------------|-------|----------|
| 期望结果 | `outcome` | `understand` (Goal Refiner) | `product_value_framework.md` |
| 用户群 | `persona` | `understand` (Goal Refiner) | — |
| 痛点 | `problem` | `understand` (Goal Refiner) | — |
| 机会/方案 | `solution` | `understand` + `map` | `product_value_framework.md` |
| 验证假设 | Outcome Mapping | `map` (Outcome Mapper) | `product_value_framework.md` |
| 评审通过 | Review Score | `review` (Reviewer Simulator) | `rubrics.md` |

## 三、产品价值映射方法（Problem → Solution → Outcome）

### 3.1 价值映射三层模型

```
Layer 1: Problem Space (问题空间)
  "用户面临什么困难？"
       │
       ▼
Layer 2: Solution Space (方案空间)
  "我们能提供什么能力？"
       │
       ▼
Layer 3: Outcome Space (结果空间)
  "成功后会看到什么变化？"
```

### 3.2 映射转换表

| Problem (用户痛点) | Solution (方案能力) | Outcome (可度量结果) |
|--------------------|--------------------|--------------------|
| 用户找不到功能入口 | 重新设计信息架构+搜索 | 功能发现率 +40% |
| 注册流程 12 步太繁琐 | 简化为 3 步核心流程 | 转化率 12%→28% |
| 数据报告手动制作耗时 | 自动生成周报 | 节省分析师 8h/周 |
| 多团队协作信息不同步 | 统一看板+实时通知 | 信息延迟 < 1min |
| 代码部署需要手动操作 | CI/CD 自动化 | 部署频率 1次/周→5次/天 |
| 文档与代码不同步 | 代码即文档 | 文档更新延迟 = 0 |

### 3.3 BOS-FS 价值转换规则

**从 Goal Refiner 到 Outcome Mapper 的传递**:

```
Goal Refiner 输出:
  {persona: "中小商家", problem: "人工盘点耗时", solution: "智能库存管理系统", outcome: "节省30%人力"}
       │
       ▼
Outcome Mapper 转换:
  Feature: "智能库存管理系统"
  Capability: "自动化库存盘点与预警"
  Outcome: "盘点时间减少70%，人力成本降低30%"
```

**转换原则**:
1. Feature 必须来自实际技术能力，不可虚构
2. Capability 回答"让用户能做什么？"
3. Outcome 必须包含可度量指标（数值/比例/时间）
4. 无法量化时标注"待验证"而非虚构数据

## 四、假设测试框架（Assumption Testing Framework）

### 4.1 四类核心假设

| 假设类型 | 关键问题 | 风险等级 | 测试方法 |
|---------|----------|---------|---------|
| **价值假设** | 用户真的关心这个问题吗？ | 🔴 高 | 用户访谈/问卷调查/落地页测试 |
| **可用性假设** | 用户会使用我们的方案吗？ | 🟡 中 | 原型测试/可用性研究/灰度发布 |
| **可行性假设** | 技术上能做到吗？ | 🟡 中 | 技术预研/POC/性能测试 |
| **商业假设** | 这个方案能盈利吗？ | 🔴 高 | 成本估算/定价实验/竞品分析 |

### 4.2 假设测试画布

```
┌─────────────────────────────────────────────┐
│ 假设测试画布                                  │
├─────────────────────────────────────────────┤
│ 假设: [用一句话描述你的假设]                   │
│ 类型: [价值/可用性/可行性/商业]                │
│ 信心度: [高/中/低]                            │
│ 如果为真: [世界会变成什么样？]                 │
│ 如果为假: [最坏结果是什么？]                   │
│ 测试方法: [如何验证？]                        │
│ 成功标准: [什么信号表明假设成立？]              │
│ 时间预算: [愿意花多少时间验证？]               │
└─────────────────────────────────────────────┘
```

### 4.3 BOS-FS 中的假设验证流程

| Pipeline Stage | 对应假设类型 | 验证机制 |
|---------------|-------------|---------|
| `understand` (Goal Refiner) | 价值假设 | 检查 persona/problem/outcome 是否来自真实用户反馈 |
| `map` (Outcome Mapper) | 可用性假设 | Feature→Outcome 转换是否合理可验证 |
| `review` (Reviewer Simulator) | 可行性假设 | 四类评审模拟检验技术/商业/产品可行性 |
| `build` (Submission Builder) | 商业假设 | 生成 risk_disclosure.md 披露商业风险 |

### 4.4 假设优先级矩阵

以「影响力 × 不确定性」排序:

```
        高不确定性
           │
  优先测试  │   高风险假设
  (探索)    │   (立即验证)
           │
 ──────────┼──────────
           │
  低风险    │   低优先级
  (已有证据)│   (暂缓)
           │
        低不确定性
```

**实操步骤**:
1. 列出所有关键假设（不超过 10 个）
2. 按影响力排序（对目标的影响程度）
3. 按不确定性排序（当前有多少证据）
4. 优先测试「高影响力 × 高不确定性」的假设
5. 在提交包中通过 `risk_disclosure.md` 记录未验证假设

## 五、持续访谈指南（Continuous Interviewing）

### 5.1 每周访谈节奏

| 维度 | 要求 |
|------|------|
| **频率** | 每周至少 1 位用户对话 |
| **形式** | 30-45 分钟一对一（视频/面谈优先） |
| **记录** | 结构化笔记：痛点、原话、行为观察 |
| **覆盖** | 轮换覆盖不同用户群/场景 |

### 5.2 访谈提问模板

**禁止问的问题**:
- ❌ "你喜欢这个功能吗？" → 用户会礼貌说"喜欢"
- ❌ "你会为这个付费吗？" → 假设性付费意愿不等于真实付费
- ❌ "如果我们有XX功能，你觉得怎么样？" → 用户对不存在的东西没有真实感受

**应该问的问题**:
- ✅ "上一次遇到这个问题是什么时候？" → 确认问题真实存在
- ✅ "当时你是怎么解决的？" → 了解现有替代方案
- ✅ "这个解决方案哪里让你不满意？" → 发现痛点细节
- ✅ "你愿意为解决这个问题放弃什么？" → 验证真实支付意愿（时间/金钱/习惯）

### 5.3 BOS-FS 访谈记录映射

访谈输出应映射到 Goal Refiner 字段:

| 访谈发现 | 映射字段 | 示例 |
|---------|---------|------|
| "我是运维，每次部署都要手动检查10个配置项" | `persona` + `problem` | persona="运维工程师", problem="部署前需手动检查10个配置项，耗时且易错" |
| "上次因为漏了一个配置导致服务挂了2小时" | `outcome` | outcome="减少部署错误，MTTR从2小时降至15分钟" |
| "我现在用Excel表格对照检查" | `solution` 灵感 | 自动化配置校验工具 |

## 六、提交前发现检查清单

在运行 BOS-FS Pipeline 之前，完成以下检查:

### 6.1 意图完整度检查

| 序号 | 检查项 | 通过标准 | 对应文件 |
|------|--------|---------|---------|
| D1 | persona 是否明确具体？ | 不是"用户"而是"中小电商企业的运维人员" | `01_goal_refiner.md` |
| D2 | problem 是否来自真实用户反馈？ | 有访谈/数据支撑，标注来源 | — |
| D3 | outcome 是否可量化？ | 包含具体数值或"待验证"标注 | `04_outcome_mapper.md` |
| D4 | 是否识别了至少 1 个关键假设？ | 填写假设测试画布（§4.2） | — |
| D5 | 是否完成至少 3 轮用户对话？ | 有访谈记录可追溯 | — |
| D6 | 是否绘制了机会树？ | 至少覆盖 2 个用户群的痛点 | §二 |
| D7 | 是否分析了竞品替代方案？ | 用户当前怎么解决这个问题？ | `product_value_framework.md` |

### 6.2 假设验证检查

| 序号 | 检查项 | 通过标准 | 状态 |
|------|--------|---------|------|
| A1 | 价值假设已验证 | 至少 3 位用户确认该问题是真实痛点 | ☐ |
| A2 | 可用性假设已验证 | 原型/DEMO 通过可用性测试 | ☐ |
| A3 | 可行性假设已验证 | 技术预研完成，无阻塞性技术风险 | ☐ |
| A4 | 商业假设已验证 | 成本/定价/竞品分析完成 | ☐ |

### 6.3 检查清单与 Pipeline 集成

- D1-D3 → 在 `understand` Stage 自动检查
- D4-D5 → 用户自行完成，Pipeline 不做强制验证
- D6 → 在 `map` Stage 中通过机会映射间接验证
- D7 → 在 `review` Stage 中由投资评审检查

**未通过检查的处理**:
- D1/D2/D3 缺失 → Goal Refiner 自动标注"未明确"或"（推断）"
- A1-A4 未验证 → 在 `risk_disclosure.md` 中作为未验证假设披露
- 建议在提交包中诚实标注"待验证"而非掩盖不确定性

## 七、发现模式速查表

### 7.1 常见发现反模式

| 反模式 | 症状 | 后果 | 修正方法 |
|--------|------|------|---------|
| **功能驱动** | 从"我们要做什么功能"出发 | 产出大量无人使用的功能 | 切换到"用户需要什么结果" |
| **一次性发现** | 项目启动时做一次用户研究 | 发现结果随市场变化失效 | 建立每周访谈习惯 |
| **验证偏差** | 只问支持预设假设的问题 | 得出错误结论 | 主动寻找反证，问"什么情况下这个方案不成立？" |
| **跳过问题空间** | 直接跳到方案设计 | 解决了一个不重要的问题 | 先用机会树探索问题空间 |
| **伪量化** | 给 outcome 编造数字 | 评审时发现数据不可信 | 标注"待验证"或引用真实数据源 |

### 7.2 高效发现模式

| 模式 | 做法 | 适用场景 |
|------|------|---------|
| **每周5分钟回顾** | 每周五花5分钟检查: 本周学到了什么关于用户的新认知？ | 所有项目 |
| **假设看板** | 维护一个假设列表，标注状态: 待验证/已验证/已推翻 | 中大型项目 |
| **用户故事地图** | 按用户旅程排列功能，标注哪些已被验证 | 产品规划阶段 |
| **最小可验证方案** | 用最少的投入验证最高风险的假设 | 早期项目/新方向 |
| **竞品替代分析** | 问"用户现在怎么解决这个问题？"而非"竞品是什么？" | 所有项目 |

## 九、机会解决方案树（OST）完整方法论

### 9.1 OST 核心结构

机会解决方案树是 Teresa Torres 发现框架的核心工具，它将「期望结果→用户机会→解决方案」以树形结构可视化。

```
期望结果 (Outcome)
"将用户注册转化率从 12% 提升到 28%"
│
├── 用户群: 新用户
│   ├── 痛点: "注册流程太长，12步"
│   │   ├── 机会: "如何将注册减少到 3 步？"
│   │   │   ├── 方案 A: 社交账号一键登录 ← 验证中
│   │   │   ├── 方案 B: 渐进式注册 ← 已验证 ✅
│   │   │   └── 方案 C: 邀请码注册 ← 已淘汰 ❌
│   │   └── 机会: "如何减少用户输入字段？"
│   │       ├── 方案 A: 智能地址补全 ← 验证中
│   │       └── 方案 B: OCR 身份证识别 ← 已淘汰 ❌
│   └── 痛点: "不清楚注册后能获得什么"
│       └── 机会: "如何让用户感知注册价值？"
│           ├── 方案 A: 注册前演示核心价值 ← 已验证 ✅
│           └── 方案 B: 免费试用期 ← 验证中
│
└── 用户群: 回归用户
    └── 痛点: "忘记密码后找回流程复杂"
        └── 机会: "如何简化密码找回？"
            └── 方案 A: 生物识别登录 ← 已验证 ✅
```

### 9.2 OST 构建规则

**树的三个层次**:

| 层次 | 内容 | 格式 | 数量限制 |
|------|------|------|---------|
| 根 (Outcome) | 期望的量化结果 | "将 X 从 A 提升到 B" | 每个树 1 个 |
| 机会 (Opportunity) | 用户/痛点/需求的组合 | "如何让用户…？" | 每个用户群 3-5 个痛点 |
| 解决方案 (Solution) | 具体的产品想法 | 可验证的假设 | 每个机会 2-4 个方案 |

**关键约束**:
- 机会层不包含解决方案（不说"如何让用户使用一键登录？"）
- 机会层聚焦用户视角（不说"如何实现 OAuth？"）
- 解决方案必须可测试（不可测试的方案不写入树）
- 每个节点标注验证状态

## 十、持续采访技术（Continuous Interviewing Techniques）

### 10.1 采访节奏设计

**每周采访节奏**:

```
周一至周五:
  周一:  筛选本周采访对象 (15 min)
  周二:  采访 1 (30-45 min)
  周三:  采访 2 (30-45 min)
  周四:  整理访谈笔记 + 更新 OST (30 min)
  周五:  团队分享发现 (15 min)

每月目标: 8-10 次用户对话
每季度目标: 24-30 次用户对话
```

### 10.2 采访类型与提问技术

**五种采访类型**:

| 类型 | 目的 | 时机 | 核心问题 |
|------|------|------|---------|
| **背景采访** | 了解用户生活/工作全貌 | 发现早期 | "描述你典型的一天" |
| **问题采访** | 深入了解具体痛点 | 机会识别阶段 | "上次遇到这个问题是什么时候？" |
| **概念测试** | 测试新方案的理解度 | 方案设计阶段 | "你看到这个方案时第一反应是什么？" |
| **可用性测试** | 测试方案的可用性 | 原型阶段 | "请完成 XX 任务" |
| **结果验证** | 验证方案是否带来预期结果 | 上线后 | "这个功能改变了你的工作方式吗？" |

### 10.3 采访提问框架

**故事采集法 (Story Collection)**:
```
不要问: "你通常怎么做 XX？" (用户会说概括性答案)
应该问: "上一次做 XX 是什么时候？" (引导回忆具体场景)

追问链条:
  "那是什么时候？"
  "当时你在哪里？"
  "还有谁在场？"
  "你当时用了什么工具？"
  "那一步之后你做了什么？"
  "最后结果怎么样？"
  "这件事对你的影响是什么？"
```

**情境锚定法 (Contextual Anchoring)**:
```
锚定具体情境:
  "假设现在是周一早上 9 点，你刚打开电脑，第一件事做什么？"
  "如果你的经理走进来问你这个问题的进展，你会怎么说？"
  "上次你向同事求助这个问题是什么时候？"

避免抽象:
  ❌ "你对这个功能有什么看法？"
  ✅ "上次你使用类似功能时，遇到了什么困难？"
```

### 10.4 采访记录到 OST 的映射流程

```
采访原始记录
    │
    ▼
Step 1: 提取关键事件和原话
    │  例: "我花了 3 天才完成注册，因为第 7 步总是报错"
    │
    ▼
Step 2: 归类到用户群和痛点
    │  用户群: 新用户
    │  痛点: 注册流程中的技术障碍
    │
    ▼
Step 3: 转化为机会问题
    │  机会: "如何让用户在注册过程中获得即时反馈？"
    │
    ▼
Step 4: 生成候选方案
    │  方案 A: 实时表单校验
    │  方案 B: 分步保存进度
    │  方案 C: 注册进度条 + 剩余步骤提示
    │
    ▼
Step 5: 更新 OST + 设计验证实验
```

## 十一、假设映射和测试优先级

### 11.1 假设映射矩阵

**假设的四维度映射**:

| 维度 | 问题 | 示例 | 验证成本 |
|------|------|------|---------|
| **用户价值假设** | 用户是否关心这个痛点？ | "新用户确实在注册时感到困扰" | 低（访谈） |
| **可用性假设** | 用户是否会使用这个方案？ | "用户会使用渐进式注册" | 中（原型测试） |
| **可行性假设** | 技术上能否实现？ | "OAuth 集成可在 2 周内完成" | 中（技术 POC） |
| **商业可行性假设** | 方案是否可持续？ | "社交登录不会影响数据安全合规" | 高（法务+安全审查） |

### 11.2 假设测试优先级排序

**ICE 评分模型**:

```
ICE Score = Impact × Confidence × Ease

Impact (影响力):      1-10 分，对 outcome 的影响程度
Confidence (信心度):  1-10 分，对假设成立的信心（1=猜的，10=有数据支撑）
Ease (简易度):        1-10 分，验证的容易程度（10=最容易）

优先级排序:
  高 ICE (70-1000)  →  立即测试
  中 ICE (20-70)    →  排队测试
  低 ICE (1-20)     →  暂缓或放弃
```

**RICE 评分模型**（扩展版）:

```
RICE Score = (Reach × Impact × Confidence) / Effort

Reach (覆盖范围):    受影响的用户数量（月度）
Impact (影响力):     0.25=最小, 0.5=低, 1=中, 2=高, 3=极高
Confidence (信心度): 0.5=低信心, 0.8=中信心, 1.0=高信心
Effort (投入):       人月数
```

### 11.3 BOS-FS 假设测试集成

```
假设 → 测试 映射表 (在 BOS-FS Pipeline 中的体现):

假设类型            Pipeline Stage         验证方式
──────────────────────────────────────────────────────
用户价值假设    →   understand (Goal Refiner)   检查 persona/problem 是否有数据支撑
可用性假设      →   map (Outcome Mapper)        Feature→Outcome 转换合理性
可行性假设      →   review (Reviewer Simulator) 技术评审模拟
商业可行性      →   build (Submission Builder)  risk_disclosure.md 风险披露
```

### 11.4 假设测试实验设计

**最小验证实验画布**:

```
┌─────────────────────────────────────────────────────┐
│ 假设测试实验画布                                      │
├─────────────────────────────────────────────────────┤
│ 假设: [一句话]                                        │
│ 如果 [这个假设成立]，那么 [可观察的现象]               │
│                                                     │
│ 实验类型: [访谈/原型/落地页/灰度/A-B]                  │
│ 样本量: [需要多少用户参与？]                          │
│ 成功标准: [什么结果表明假设成立？]                     │
│ 失败标准: [什么结果表明假设不成立？]                   │
│ 时间预算: [愿意花多少时间？]                          │
│ 成本预算: [愿意花多少资源？]                          │
│                                                     │
│ 决策规则:                                            │
│  ✅ 通过 → 进入下一阶段                               │
│  ❌ 失败 → 放弃或调整假设                              │
```

## 十三、结果导向 vs 产出导向的度量

### 13.1 产出指标 vs 结果指标对比

| 维度 | 产出指标 (Output) | 结果指标 (Outcome) |
|------|------------------|-------------------|
| **定义** | 做了什么/交付了什么 | 产生了什么影响/改变了什么 |
| **示例** | 完成了 5 个功能 | 转化率提升了 15% |
| **时间窗口** | 当次迭代 | 持续追踪 |
| **可控性** | 团队完全可控 | 受多因素影响 |
| **价值判断** | 无法判断是否成功 | 直接反映价值 |
| **激励效果** | 激励多做功能 | 激励做对的功能 |

### 13.2 结果指标设计框架

**OKR 中的结果导向**:

```
Objective (定性): "让新用户注册体验更顺畅"
  │
  ├── Key Result 1: 注册转化率从 12% → 28%
  ├── Key Result 2: 注册平均耗时从 5 min → 2 min
  ├── Key Result 3: 注册放弃率从 65% → 30%
  └── Key Result 4: 注册后 7 日留存从 40% → 60%
```

**北星指标 (North Star Metric)**:

| 产品类型 | 北星指标示例 | 说明 |
|---------|-------------|------|
| 社交产品 | 日活跃用户 × 平均互动次数 | 反映真实参与度 |
| 电商平台 | 周购买用户数 | 反映商业价值 |
| SaaS 工具 | 周活跃团队数 | 反映团队协作价值 |
| 内容平台 | 周内容消费时长 | 反映内容吸引力 |

### 13.3 BOS-FS 中的结果度量

**BOS-FS 输出指标（产出 → 需避免）**:
- ❌ Pipeline 运行次数
- ❌ 生成的文件数量
- ❌ 评审模拟运行次数

**BOS-FS 结果指标（结果 → 应关注）**:
- ✅ 提交包被接受率
- ✅ 评审一次性通过率
- ✅ 价值陈述完整度（Feature→Outcome 转换覆盖率）
- ✅ 用户/评审方反馈满意度

### 13.5 从产出转向结果的实践步骤

```
Step 1: 识别当前的产出指标
  "我们目前度量什么？" → 列出所有产出指标

Step 2: 对每个产出指标问 "So What?"
  "完成了 5 个功能" → So What? → "用户能做什么以前不能做的？"
  → 转化为结果指标

Step 3: 定义 1-3 个结果指标
  选择最核心的结果指标作为北星指标
  补充 2-3 个支撑性结果指标

Step 4: 建立结果追踪机制
  在 BOS-FS Pipeline 中采集结果数据
  定期（每周/双周）回顾结果趋势

Step 5: 调整决策标准
  评审标准从"功能是否完成"转向"结果是否达成"
  在 risk_disclosure.md 中记录结果偏差
```
