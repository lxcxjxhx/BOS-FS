# 流水线综合评分标准

## 评分模型

### 阶段权重
| 阶段 | 权重 | 评分标准 |
|------|------|----------|
| Goal Refinement | 15% | persona/problem/solution/outcome 完整度 |
| Outcome Mapping | 15% | Feature→Capability→Outcome 转换质量 |
| README Refactor | 25% | 五段式结构完整度、价值公式应用 |
| Review Simulation | 25% | 四类评审平均通过率 |
| Submission Build | 20% | 提交包组件完整性、一致性 |

### 评分细则

#### Goal Refinement (0-10)
| 分数 | 标准 |
|------|------|
| 8-10 | 四个字段完整且具体，有量化指标 |
| 5-7 | 四个字段基本完整，部分泛化 |
| 2-4 | 字段缺失或过于模糊 |
| 0-1 | 完全无法提取有效意图 |

#### Outcome Mapping (0-10)
| 分数 | 标准 |
|------|------|
| 8-10 | 所有 Feature 都转换为可量化 Outcome |
| 5-7 | 主要 Feature 有转换，部分泛化 |
| 2-4 | 仅有 Capability 层转换 |
| 0-1 | 无有效转换 |

#### README Refactor (0-10)
| 分数 | 标准 |
|------|------|
| 8-10 | 五段式完整、架构图清晰、指标可量化 |
| 5-7 | 结构基本完整，部分缺失细节 |
| 2-4 | 仅有部分段落，结构混乱 |
| 0-1 | 未重构或完全无效 |

#### Review Simulation (0-10)
| 分数 | 标准 |
|------|------|
| 8-10 | 四类评审平均通过率 > 70% |
| 5-7 | 平均通过率 50-70% |
| 2-4 | 平均通过率 30-50% |
| 0-1 | 平均通过率 < 30% |

#### Submission Build (0-10)
| 分数 | 标准 |
|------|------|
| 8-10 | 所有组件完整生成，内容一致 |
| 5-7 | 核心组件完整，少量缺失 |
| 2-4 | 仅部分组件生成 |
| 0-1 | 提交包未生成或严重不一致 |

### 综合评分计算
`pipeline_score = Σ(stage_score × weight)`

### 通过阈值
| 等级 | 分数 | 含义 |
|------|------|------|
| S | ≥ 9.0 | 可直接提交 |
| A | 8.0-8.9 | 建议微调后提交 |
| B | 6.0-7.9 | 需要改进后重投 |
| C | 4.0-5.9 | 大幅修改后重投 |
| D | < 4.0 | 需要重新规划项目定位 |

### 改进建议生成规则
- Goal Refinement < 5 → 重新提炼项目意图
- Outcome Mapping < 5 → 补充价值转换
- README Refactor < 5 → 重构 README 结构
- Review Simulation < 5 → 处理评审拒绝理由
- Submission Build < 5 → 补齐提交包组件
