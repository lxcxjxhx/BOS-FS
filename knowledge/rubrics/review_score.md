# 评审评分标准

## 评分维度

| 维度 | 分值 | 说明 |
|------|------|------|
| 技术深度 | 0-10 | 技术选型的合理性、架构设计的复杂度、代码质量 |
| 创新性 | 0-10 | 方案的原创性、与现有方案的差异化程度 |
| 实用价值 | 0-10 | 解决实际问题的能力、可落地性、市场潜力 |
| 文档质量 | 0-10 | README 完整性、API 文档、注释、测试覆盖率 |

## 评分细则

### 技术深度 (0-10)

| 分数 | 标准 |
|------|------|
| 9-10 | 架构设计精良，技术选型有深度论证，代码质量极高 |
| 7-8 | 技术方案合理，有一定的技术难点攻克 |
| 5-6 | 基础技术实现，无明显技术亮点 |
| 3-4 | 技术实现粗糙，存在设计缺陷 |
| 0-2 | 无法运行或技术严重过时 |

### 创新性 (0-10)

| 分数 | 标准 |
|------|------|
| 9-10 | 行业首创或显著突破，有专利/论文支撑 |
| 7-8 | 有明显创新点，与竞品形成差异化 |
| 5-6 | 微创新，在现有方案上有所改进 |
| 3-4 | 基本是已有方案的复制 |
| 0-2 | 无任何创新 |

### 实用价值 (0-10)

| 分数 | 标准 |
|------|------|
| 9-10 | 痛点明确，已有用户验证，商业路径清晰 |
| 7-8 | 解决真实问题，有潜在市场需求 |
| 5-6 | 有一定应用场景，但需求不够强烈 |
| 3-4 | 伪需求或市场极小 |
| 0-2 | 无实际应用场景 |

### 文档质量 (0-10)

| 分数 | 标准 |
|------|------|
| 9-10 | 文档完备，可独立支撑二次开发和部署 |
| 7-8 | 主要文档齐全，偶有遗漏 |
| 5-6 | 有基本文档，但不够详细 |
| 3-4 | 文档严重缺失 |
| 0-2 | 几乎无文档 |

## 总分解读

| 总分 | 等级 | 解读 |
|------|------|------|
| ≥ 30 | ✅ 通过 | 项目质量达标，可进入下一轮 |
| 24-29 | ⚠️ 有条件通过 | 需在指定方向改进后复审 |
| < 24 | ❌ 不通过 | 项目不满足基本要求 |

**通过阈值：总分 ≥ 30/40**
