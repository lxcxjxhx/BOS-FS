# 数据密集型应用架构模式

> 来源: Designing Data-Intensive Applications — The Big Ideas Behind Reliable, Scalable, and Maintainable Systems (Martin Kleppmann)
> 映射: BOS-FS 五层架构 → Runtime Layer + Engine Layer + Governance Layer
> 用途: 提供可靠性、可伸缩性、可维护性的系统设计模式，指导 BOS-FS 运行时引擎在数据流处理、分布式一致性、事务管理和演化式架构方面的设计决策。

---

## 一、可靠性模式（Fault Tolerance, Replication, Transactions）

### 1.1 可靠性三要素

DDIA 定义可靠性包含三个维度：

| 维度 | 定义 | BOS-FS 场景 | 实现方式 |
|------|------|------------|---------|
| **容错 (Fault Tolerance)** | 系统在部分组件故障时仍能继续工作 | Pipeline Stage 故障 | Stage 异常隔离 + 重试机制 |
| **持久性 (Durability)** | 数据不因系统故障而丢失 | Pipeline 运行状态 | 文件系统持久化 + JSON 日志 |
| **低延迟 (Low Latency)** | 系统在正常和故障情况下都能快速响应 | Pipeline 执行时间 | 缓存 + 并发优化 |

### 1.2 容错模式

**硬件故障 → 软件容错的转换**:
```
硬件不可靠 (故障是常态)
    │
    ▼
软件容错层 (容忍故障)
    │
    ├── 冗余 (Replication)     → PipelineContext 多副本
    ├── 隔离 (Isolation)       → Stage 异常不传播
    ├── 超时 (Timeout)         → Stage 执行超时检测
    ├── 重试 (Retry)           → Stage 失败自动重试
    └── 降级 (Degradation)     → 部分 Stage 跳过
```

**容错设计模式速查**:

| 模式 | 场景 | 实现 | BOS-FS 应用 |
|------|------|------|------------|
| **健康检查** | 检测组件是否正常 | 心跳/探针 | Stage `validate_input()` 前置检查 |
| **超时控制** | 防止无限等待 | 定时器 | PipelineConfig `stage_timeout` |
| **重试策略** | 临时故障恢复 | 指数退避 | `execute()` 重试计数器 |
| **熔断器** | 防止级联故障 | 失败计数 → 开路 | 连续 N 次失败 → 跳过 Stage |
| **隔离舱** | 限制故障范围 | 资源隔离 | Stage try/except 隔离 |
| **优雅降级** | 部分功能不可用时 | 功能开关 | FeatureFlag 禁用问题 Stage |

### 1.3 复制模式

**Leader-Follower 复制**:
```
适用场景: 读多写少的数据系统

  Leader (写入)
    │
    ├──▶ Follower 1 (读取)
    ├──▶ Follower 2 (读取)
    └──▶ Follower 3 (读取)

同步复制: 强一致性，但写入延迟高
异步复制: 写入快，但可能丢数据
半同步: 平衡一致性与性能
```

**BOS-FS 中的复制模式**:
| 复制场景 | 模式 | 一致性模型 |
|---------|------|-----------|
| knowledge/ 文件分发 | 异步复制 | 最终一致性 |
| submission_bundle/ 输出 | 同步写入 | 强一致性 |
| PipelineContext 共享数据 | Leader-Follower | 半同步 |

### 1.4 事务模式

**ACID 属性**:
| 属性 | 定义 | BOS-FS 映射 |
|------|------|------------|
| **Atomicity (原子性)** | 事务全部成功或全部失败 | Pipeline 全 Stage 成功或回滚 |
| **Consistency (一致性)** | 事务前后数据保持约束 | Stage 输入/输出 schema 验证 |
| **Isolation (隔离性)** | 并发事务互不干扰 | Pipeline 单实例运行，天然隔离 |
| **Durability (持久性)** | 提交后数据不丢失 | 文件系统写入 |

**事务隔离级别与异常**:

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 性能 |
|---------|------|-----------|------|------|
| Read Uncommitted | 可能 | 可能 | 可能 | 最高 |
| Read Committed | ❌ | 可能 | 可能 | 高 |
| Repeatable Read | ❌ | ❌ | 可能 | 中 |
| Serializable | ❌ | ❌ | ❌ | 最低 |

**BOS-FS 事务策略**: Pipeline 顺序执行 → 天然 Serializable 隔离级别（无并发竞争）。

### 1.5 可靠性检查清单

| 序号 | 检查项 | 通过标准 | 说明 |
|------|--------|---------|------|
| R1 | 每个 Stage 异常隔离 | try/except 包裹 | 单 Stage 失败不影响其他 |
| R2 | 运行状态持久化 | JSON 日志文件 | 支持 resume 恢复 |
| R3 | 输入数据校验 | schema 验证 | 拒绝无效输入 |
| R4 | 超时控制 | stage_timeout 配置 | 防止无限等待 |
| R5 | 重试机制 | max_retries 配置 | 临时故障自动恢复 |
| R6 | 降级策略 | fallback 实现 | 非关键 Stage 可跳过 |
| R7 | 运行监控 | 事件日志可查询 | PipelineContext 完整记录 |

## 二、可伸缩性模式（Partitioning, Replication, Load Balancing）

### 2.1 可伸缩性描述方式

| 描述维度 | 内容 | BOS-FS 场景 |
|---------|------|------------|
| **负载描述** | 用什么参数衡量系统压力？ | 输入描述大小 / 知识文件数量 / Pipeline 并发数 |
| **性能指标** | 用什么衡量系统表现？ | Pipeline 总耗时 / 内存占用 / Stage 延迟 |
| **伸缩策略** | 增加资源后性能如何变化？ | 当前：单进程 → 未来：多进程/分布式 |

### 2.2 分区模式 (Partitioning)

**分区策略**:

```
分区目标: 将大型数据集分散到多个节点，提高并行处理能力

┌─────────────────────────────────────────────────┐
│              Partitioning Strategies               │
│                                                    │
│  1. 键值范围分区 (Key-Range Partitioning)          │
│     • 按字母/数字范围分片                          │
│     • 适合范围查询                                 │
│     • 可能导致热点                                 │
│                                                    │
│  2. 哈希分区 (Hash Partitioning)                   │
│     • hash(key) % N 决定分区                       │
│     • 数据分布更均匀                               │
│     • 不支持高效范围查询                           │
│                                                    │
│  3. 一致性哈希 (Consistent Hashing)                │
│     • 节点增减时最小化数据迁移                      │
│     • 虚拟节点减少不均匀                           │
│                                                    │
└─────────────────────────────────────────────────┘
```

**BOS-FS 中的分区映射**:

| 数据域 | 分区方式 | 分区键 | 目的 |
|--------|---------|--------|------|
| knowledge/ 目录 | 按目录分层 | 知识域 (intent/adoption/runtime) | 组织清晰，易于查找 |
| submission_bundle/ 文件 | 按 run_id 分区 | 每次运行的 UUID | 隔离每次输出 |
| PipelineContext | 按运行实例分区 | run_id | 并发运行互不干扰 |

### 2.3 负载均衡模式

**负载均衡算法对比**:

| 算法 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| 轮询 (Round Robin) | 依次分配请求 | 简单公平 | 不考虑负载差异 | 同构节点 |
| 加权轮询 | 按权重分配 | 支持异构节点 | 权重需手动设置 | 异构节点 |
| 最少连接 | 分配给连接最少的节点 | 自动适应负载 | 需要维护连接计数 | 请求耗时差异大 |
| 一致性哈希 | 相同 key → 相同节点 | 支持缓存亲和 | 节点变更影响部分数据 | 有状态服务 |

**BOS-FS 负载模型**: 当前为单进程，负载集中在 AI 处理阶段。未来分布式架构下：
```
负载均衡器
    │
    ├──▶ Worker 1 (Intent Parser)
    ├──▶ Worker 2 (Outcome Mapper)
    ├──▶ Worker 3 (Reviewer Simulator)
    └──▶ Worker 4 (Submission Builder)

调度策略: 按 Stage 类型路由 → 最少连接负载均衡
```

### 2.4 可伸缩性检查清单

| 序号 | 检查项 | 当前状态 | 改进方向 |
|------|--------|---------|---------|
| S1 | 数据分区策略 | 目录分层 | 支持水平分区 |
| S2 | 并发处理 | 单线程顺序 | 未来 DAG 并行 |
| S3 | 内存使用 | 全量加载 | 流式处理大文件 |
| S4 | 缓存策略 | Stage 级缓存 | 多级缓存 (L1/L2) |
| S5 | I/O 优化 | 同步读写 | 异步 I/O + 批量写入 |

## 三、可维护性模式（Evolvability, Observability, Testability）

### 3.1 演化式设计原则

DDIA 核心观点: **系统最重要的特性是能够随时间演化**。

| 原则 | 内容 | BOS-FS 实践 |
|------|------|------------|
| **抽象泄漏最小化** | 隐藏实现细节 | Stage Protocol 接口抽象 |
| **操作友好性** | 运维简单 | 本地运行，零配置 |
| **简单性** | 降低理解成本 | 7 Stage 清晰分工 |
| **可演进性** | 支持渐进式变更 | Custom Stage 注册机制 |

### 3.2 可观测性三层模型

```
┌─────────────────────────────────────────────────────────┐
│                  Observability Stack                       │
│                                                          │
│  ┌───────────────┐    ┌───────────────┐    ┌────────────┐│
│  │    Metrics     │    │     Logs      │    │   Traces   ││
│  │    (指标)      │    │    (日志)     │    │   (追踪)   ││
│  │               │    │               │    │            ││
│  │ • Pipeline耗时 │    │ • Stage 输入  │    │ • run_id   ││
│  │ • Stage 成功率 │    │ • Stage 输出  │    │ • 完整链路  ││
│  │ • 缓存命中率   │    │ • 异常信息    │    │ • 耗时分解  ││
│  │ • 内存使用     │    │ • 运行状态    │    │ • 依赖关系  ││
│  └───────────────┘    └───────────────┘    └────────────┘│
│                                                          │
│  Metrics: 回答"系统有多健康？" (聚合数据)                  │
│  Logs:     回答"系统发生了什么？" (原始事件)                │
│  Traces:   回答"请求经过了哪些组件？" (调用链)              │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

**BOS-FS 可观测性实现**:

| 可观测性维度 | 数据来源 | 采集方式 | 使用场景 |
|-------------|---------|---------|---------|
| Metrics | PipelineContext | 运行后统计 | 效能仪表盘 / 趋势分析 |
| Logs | Stage 输出 | JSON 日志文件 | 故障排查 / 审计 |
| Traces | Pipeline run_id | 贯穿全链路的 UUID | 端到端追踪 |

### 3.3 演化式架构实践

**Schema 演化策略**:

```
向后兼容 (Backward Compatible):
  新版本能读取旧版本数据

向前兼容 (Forward Compatible):
  旧版本能读取新版本数据（忽略新字段）

BOS-FS 策略:
  • PipelineContext 使用 JSON → 天然支持添加新字段
  • Stage input_schema 只检查必需字段 → 可选字段可增量添加
  • 禁止删除必需字段 → 保证向后兼容
  • 新增字段标记为 optional → 保证向前兼容
```

**API 版本管理**:
```
v1.0: 基础 Pipeline (6 Stage)
v1.1: 添加 reject_analyzer
v1.2: 添加知识文件索引
v2.0: (计划) DAG Pipeline + 并行执行

版本迁移策略:
  • 大版本: 不兼容变更，需要用户显式升级
  • 小版本: 向后兼容，自动升级
```

### 3.4 可维护性检查清单

| 序号 | 检查项 | 通过标准 | 说明 |
|------|--------|---------|------|
| M1 | 新增 Stage 是否 < 1 天工作量？ | 注册即用 | 验证可插拔性 |
| M2 | 新人能否 < 1 小时理解架构？ | 文档 + 代码清晰 | 验证简单性 |
| M3 | Schema 变更是否向后兼容？ | 旧数据可读取 | 验证可演进性 |
| M4 | 是否有完整的运行日志？ | 每次运行可追溯 | 验证可观测性 |
| M5 | 是否有架构决策记录？ | ADR 文档 | 验证决策可追溯 |

## 四、数据流模式（Batch Processing, Stream Processing, Lambda/Kappa）

### 4.1 批处理模式 (Batch Processing)

**Unix 哲学在数据流中的应用**:
```
输入 → [过滤] → [转换] → [聚合] → 输出

BOS-FS Pipeline 遵循相同模式:
  项目描述 → [Goal Refiner 过滤] → [Outcome Mapper 转换]
           → [Submission Builder 聚合] → 提交包
```

**批处理关键特性**:

| 特性 | 定义 | BOS-FS 实现 |
|------|------|------------|
| **确定性** | 相同输入 → 相同输出 | cache_key 哈希保证 |
| **容错** | 失败可重跑 | resume 机制 |
| **幂等** | 多次执行效果相同 | 输出文件覆盖 |
| **排序无关** | 输入顺序不影响输出 | 标准化处理 |

### 4.2 流处理模式 (Stream Processing)

**流处理与批处理对比**:

| 维度 | 批处理 | 流处理 |
|------|--------|--------|
| 输入 | 固定大小的数据集 | 持续到达的事件流 |
| 触发 | 定时或手动触发 | 事件到达即时触发 |
| 延迟 | 高（分钟~小时） | 低（毫秒~秒） |
| 输出 | 完整结果 | 增量结果 |
| 适用 | 离线分析、报告生成 | 实时监控、告警 |

**BOS-FS 流处理场景**:
```
事件流: 用户交互 → Pipeline 触发

  [文件变更] ──▶ FileSystemWatcher
                    │
                    ▼
              Pipeline Trigger
                    │
                    ├──▶ 自动运行 Pipeline
                    ├──▶ 输出更新到 submission_bundle/
                    └──▶ 通知用户
```

### 4.3 Lambda 架构

```
                    ┌───────────────┐
                    │   查询层       │
                    │ (Serving)     │
                    └───┬───────┬───┘
                        │       │
         ┌──────────────┘       └──────────────┐
         ▼                                     ▼
┌─────────────────┐                   ┌─────────────────┐
│    批处理层      │                   │    实时层        │
│  (Batch Layer)   │                   │  (Speed Layer)   │
│                 │                   │                 │
│ • 全量数据       │                   │ • 增量数据       │
│ • 高延迟         │                   │ • 低延迟         │
│ • 高精度         │                   │ • 近似结果       │
│ • Hadoop/Spark  │                   │ • Storm/Flink   │
└─────────────────┘                   └─────────────────┘
         │                                     │
         └──────────────┐               ┌──────┘
                        ▼               ▼
                  ┌─────────────────────────┐
                  │      合并查询结果        │
                  └─────────────────────────┘

Lambda 架构的问题:
  • 维护两套代码 (批处理 + 实时)
  • 结果合并复杂
  • 调试困难
```

### 4.4 Kappa 架构

```
                    ┌───────────────┐
                    │   查询层       │
                    └───────┬───────┘
                            │
                            ▼
                  ┌─────────────────────┐
                  │     流处理层          │
                  │  (Stream Layer)      │
                  │                     │
                  │ • 统一处理所有数据   │
                  │ • 可重放历史数据     │
                  │ • 单一代码库         │
                  └─────────┬───────────┘
                            │
                            ▼
                  ┌─────────────────────┐
                  │     日志存储          │
                  │  (Append-Only Log)   │
                  │  (Kafka/Event Store) │
                  └─────────────────────┘

Kappa 架构的优势:
  • 单一代码路径
  • 通过重放实现"批处理"
  • 简化运维
```

**BOS-FS 架构演进路线**:
```
当前 (v0.11+):  类批处理模式 (每次运行处理完整输入)
中期 (v1.0):   事件驱动 + 类 Kappa (事件日志 + 重放)
远期 (v2.0+):  完全流式 (FileSystemWatcher → 实时处理)
```

### 4.5 数据流模式选择框架

```
决策树: 选择批处理还是流处理？

输入是连续的？
  ├── 否 → 批处理 (当前 BOS-FS)
  └── 是 → 需要低延迟？
              ├── 否 → 微批处理 (定时触发)
              └── 是 → 流处理 (事件驱动)

BOS-FS 当前选择: 批处理 + 定时触发
原因:
  • 用户显式触发 Pipeline 运行
  • 不需要实时响应
  • 处理逻辑简单
  • 维护成本低
```

## 五、分布式系统一致性模型

### 5.1 CAP 定理与取舍

```
CAP 定理: 分布式系统最多同时满足以下三个属性中的两个

  Consistency (一致性) ────┐
                           │
  Availability (可用性) ────┼── 只能选两个
                           │
  Partition Tolerance (分区容错) ─┘

实际选择:
  • CP: 保证一致性和分区容错 → 放弃可用性 (ZooKeeper)
  • AP: 保证可用性和分区容错 → 放弃一致性 (DynamoDB)
  • CA: 理论上不可能 (网络分区总会发生)
```

**BOS-FS 的一致性选择**: CA → 单机系统，不存在网络分区问题。天然保证一致性和可用性。

### 5.2 一致性模型对比

| 模型 | 定义 | 读保证 | 写保证 | 性能 | 适用场景 |
|------|------|--------|--------|------|---------|
| **强一致性** | 写操作完成后，所有读都能看到最新值 | 总是最新 | 同步等待所有副本 | 最低 | 金融交易 |
| **顺序一致性** | 所有操作按全局顺序执行 | 全局顺序一致 | 按序执行 | 低 | 日志系统 |
| **因果一致性** | 因果相关的操作保持顺序 | 因果顺序 | 因果传递 | 中 | 社交网络 |
| **最终一致性** | 最终所有副本收敛到相同值 | 可能读到旧值 | 异步复制 | 最高 | CDN/DNS |

### 5.3 分布式事务模式

**两阶段提交 (2PC)**:
```
Phase 1: Prepare
  Coordinator → 询问所有参与者: "能提交吗？"
  参与者 → 回复: Yes / No

Phase 2: Commit / Abort
  如果全部 Yes → Coordinator → "提交"
  如果有 No   → Coordinator → "中止"

问题: 阻塞协议，Coordinator 故障会导致参与者挂起
```

**Saga 模式**:
```
事务分解为多个子事务，每个子事务有对应的补偿操作

  T1 → T2 → T3 → ... → Tn
  │    │    │           │
  C1   C2   C3          Cn  (补偿操作)

如果 Ti 失败:
  执行 Ci, Ci-1, ..., C1 补偿

BOS-FS 中的 Saga 映射:
  Pipeline Stage 链 = Saga 事务链
  Stage 失败 = 子事务失败
  resume = 补偿后重新执行
```

### 5.4 共识算法概览

| 算法 | 类型 | 容错 | 性能 | 使用场景 |
|------|------|------|------|---------|
| **Paxos** | 共识 | f 个故障需要 2f+1 节点 | 中等 | 理论基准 |
| **Raft** | 共识 (Leader-based) | f 个故障需要 2f+1 节点 | 较好 | etcd/Consul |
| **ZAB** | 共识 (ZooKeeper) | f 个故障需要 2f+1 节点 | 较好 | ZooKeeper |
| **Gossip** | 反熵传播 | 无严格容错保证 | 高 | Cassandra |

**BOS-FS 共识**: 单机系统，不需要分布式共识。Pipeline 顺序执行天然保证顺序一致性。

### 5.5 一致性实践检查清单

| 序号 | 检查项 | 当前状态 | 未来分布式场景 |
|------|--------|---------|---------------|
| C1 | Pipeline 状态一致性 | ✅ 强一致 | 需要分布式锁 |
| C2 | 知识文件版本一致 | ✅ Git 保证 | 需要 CRDT/OT |
| C3 | 输出文件一致性 | ✅ 原子写入 | 需要 2PC/Saga |
| C4 | 缓存一致性 | ✅ Stage 级缓存 | 需要缓存失效策略 |
| C5 | 并发安全 | ✅ 单进程 | 需要锁/原子操作 |

## 七、架构决策框架

### 7.1 数据系统选型框架

```
决策流程: 选择合适的数据处理模式

1. 数据量有多大？
   ├── 小 (< 1GB) → 内存处理 / JSON 文件 ← BOS-FS 当前
   ├── 中 (1-100GB) → SQLite / 文件索引
   └── 大 (> 100GB) → 分布式存储

2. 需要多快的响应？
   ├── 秒级 → 批处理 ← BOS-FS 当前
   ├── 亚秒级 → 内存缓存 + 批处理
   └── 毫秒级 → 流处理

3. 数据如何变化？
   ├── 静态 (一次性处理) → 批处理
   ├── 低频更新 (天/周) → 定时批处理
   └── 高频更新 (秒/分) → 流处理 ← BOS-FS 未来

4. 需要多强的一致性？
   ├── 强一致 → 事务 / 两阶段提交
   ├── 最终一致 → 异步处理 ← BOS-FS 当前
   └── 不需要一致 → 无状态处理
```

### 7.2 BOS-FS 架构决策记录

**ADR-001: 选择文件系统作为存储层**
- **背景**: BOS-FS 需要存储 Pipeline 运行状态和知识文件
- **决策**: 使用文件系统 (JSON + Markdown) 而非数据库
- **权衡**:
  - 优势: 零依赖、易调试、人类可读、版本控制友好
  - 劣势: 查询能力弱、并发控制复杂
- **后果**: 当前规模足够，未来可能需要嵌入式数据库

**ADR-002: 选择顺序 Pipeline 而非 DAG**
- **背景**: Stage 执行顺序固定，暂不需要复杂依赖
- **决策**: 顺序 Pipeline + 未来支持 DAG
- **权衡**:
  - 优势: 简单、易理解、易调试
  - 劣势: 不支持并行、灵活性有限
- **后果**: 当前性能足够，v1.0+ 引入 DAG 提升并发

### 7.3 技术债务管理

| 债务类型 | 当前状态 | 影响 | 偿还时机 |
|---------|---------|------|---------|
| 单进程架构 | 当前瓶颈在 AI 处理 | 低 | 当 AI 处理 < 1 秒时 |
| 无结构化查询 | knowledge/ 文件只能 grep | 中 | 当知识文件 > 50 个时 |
| 无并发控制 | 不支持同时运行 | 低 | 当需要并行运行时 |
| 无数据压缩 | JSON 文件体积 | 低 | 当日志文件 > 100MB 时 |
