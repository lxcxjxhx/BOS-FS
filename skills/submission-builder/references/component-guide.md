# 提交包组件指南

> 来源: [SKILL.md](../SKILL.md)

本文件定义 submission-builder 的 8 个标准组件的用途、格式、必备章节、最佳实践与常见错误。

---

## 1. README.md

### 用途
项目的门面文档，评审第一眼看到的内容，决定评审对项目的第一印象。

### 格式
标准 Markdown 文件。

### 必备章节

| 章节 | 内容 | 字数建议 |
|------|------|----------|
| What | 项目是什么，一句话定义 | ≤30 字 |
| Why | 为什么做，解决的痛点 | 100-200 字 |
| How | 核心实现方式/技术栈 | 100-200 字 |
| Result | 量化成果/指标 | 50-100 字 |
| Next | 下一步计划 | 50-100 字 |

### 最佳实践
- 首段 30 秒内能说清项目价值
- 使用图标或徽章增强可读性（可选）
- 包含快速开始链接（指向 demo_guide.md）
- Result 必须有量化数据

### 常见错误
- ❌ 只有功能列表，无价值主张
- ❌ 技术术语堆砌，非技术人员无法理解
- ❌ 无量化结果（"提升效率"→应为"提升 40% 编译速度"）
- ❌ 缺失 Next 章节，无发展预期

## 2. demo_guide.md（演示指南）

### 用途
让评审能在 5 分钟内完成安装→运行→验证的端到端体验。

### 格式
标准 Markdown 文件。

### 必备章节

| 章节 | 内容 | 要求 |
|------|------|------|
| 前置条件 | 环境要求/依赖清单 | 明确操作系统、版本、依赖包 |
| 安装 | 安装步骤 | 逐条命令，可直接复制粘贴 |
| 配置 | 配置示例（含注释） | 每个配置项解释用途和默认值 |
| 运行 | 启动命令和预期输出 | 提供启动后的验证信号 |
| 验证 | 功能→预期结果对照表 | 每个核心功能对应一个验证用例 |

### 最佳实践
- 所有命令可直接复制执行，无需额外上下文
- 提供常见报错及解决方案（FAQ 式附录）
- 使用代码块标注 shell/终端输出
- 安装步骤 ≤ 5 步

### 常见错误
- ❌ 依赖描述模糊（"需要安装一些库"）
- ❌ 缺少预期输出，用户无法判断是否成功
- ❌ 配置项无注释或默认值说明
- ❌ 验证步骤缺失或不明确

## 3. introduction.md（项目简介）

### 用途
一份可在 2 分钟内读完的精炼 Pitch，用于快速评估项目价值。

### 格式
标准 Markdown 文件。

### 必备章节

| 章节 | 内容 | 字数限制 |
|------|------|----------|
| 一句话 | 15 字以内的核心价值主张 | ≤15 字 |
| 问题 | 谁 + 什么问题 + 多大影响 | ≤100 字 |
| 方案 | 关键创新点 | ≤100 字 |
| 差异化 | 与竞品 2-3 点实质性区别 | ≤100 字 |
| 指标 | 量化收益/成果 | ≤50 字 |
| 路线图 | 近期/中期/远期里程碑 | ≤100 字 |

### 最佳实践
- 一句话价值主张必须 ≤ 15 字，这是质量门
- 问题描述必须包含目标用户和影响规模
- 差异化必须基于可验证事实，非主观判断

### 常见错误
- ❌ 一句话超过 15 字
- ❌ 问题描述无目标用户或影响规模
- ❌ 差异化是"我们做得更好"等空话
- ❌ 指标无具体数字或测量方法

## 4. screenshots_guide.md（截图指南）

### 用途
规范关键界面的截图要求和说明，让评审通过视觉快速理解项目。

### 格式
标准 Markdown 文件。

### 必备章节

| 章节 | 内容 | 要求 |
|------|------|------|
| 截图规范 | 分辨率/格式/命名规则 | 推荐 1920×1080, PNG, 语义化文件名 |
| 关键界面 | ≥3 个关键界面的截图说明 | 每个界面标注"截图位置+要展示的内容" |
| 标注指南 | 截图上应标注的关键元素 | 用箭头/框/文字突出核心功能区域 |

### 最佳实践
- 覆盖登录/首页/核心功能 3 个最少场景
- 截图文件名使用 `01-login.png`、`02-dashboard.png` 格式
- 每个截图附 1-2 句说明"这张图展示什么"

### 常见错误
- ❌ 少于 3 个关键界面
- ❌ 截图模糊或分辨率过低
- ❌ 无截图说明，评审不知道看什么
- ❌ 包含敏感信息（密钥/内部 URL）未打码

## 5. FAQ.md

### 用途
预判评审和用户的典型问题，减少沟通成本。

### 格式
标准 Markdown Q&A 格式。

### 必备主题（≥8 个问答）

| # | 主题 | 示例问题 |
|---|------|----------|
| 1 | 问题定义 | 这个项目要解决什么问题？ |
| 2 | 竞品差异 | 与 [竞品] 有什么不同？ |
| 3 | 使用方法 | 如何使用核心功能？ |
| 4 | 技术基础 | 基于什么技术栈？为什么选择？ |
| 5 | 扩展性 | 如何扩展/自定义？ |
| 6 | 数据存储 | 数据存储在哪里？安全吗？ |
| 7 | 安全性 | 有哪些安全措施？ |
| 8 | 性能 | 性能表现如何？有基准测试吗？ |
| 9 | 二次开发 | 如何进行二次开发？ |
| 10 | 规划 | 未来有什么计划？ |

### 最佳实践
- 每个回答 2-4 句，简洁直接
- 包含具体数据或引用支撑回答
- 对"不知道"的回答诚实标注"待定"

### 常见错误
- ❌ 回答超过 5 句，过于冗长
- ❌ 回避问题或含糊其辞
- ❌ 少于 8 个问答
- ❌ 回答与项目实际情况不符

## 6. risk_disclosure.md（风险披露）

### 用途
主动披露已知风险，建立评审信任度。**不可省略**。

### 格式
标准 Markdown 文件，包含三张风险表。

### 必备章节

| 类别 | 最少条目 | 表格格式 |
|------|----------|----------|
| 技术风险 | ≥1 | 风险名 \| 描述 \| 影响(高/中/低) \| 缓解措施 |
| 市场风险 | ≥1 | 同上 |
| 合规风险 | ≥1 | 同上 |

### 最佳实践
- 主动披露比评审发现好 100 倍
- 每个风险必须有对应的缓解措施
- 影响等级必须诚实评估

### 常见错误
- ❌ 省略风险披露文件
- ❌ 填写"无风险"（任何项目都有风险）
- ❌ 风险描述过于笼统
- ❌ 有识别无缓解措施

## 7. trust_statement.md（信任声明）

### 用途
声明项目在技术、商业、工程三个维度的可信度，附可验证的引用。

### 格式
标准 Markdown 文件。

### 必备章节

| 章节 | 内容 | 要求 |
|------|------|------|
| 技术可信度 | 基于的权威框架/标准 | 必须附来源链接和具体条款 |
| 商业可信度 | 对标模式/市场验证 | 必须有数据或可查证引用 |
| 工程可信度 | 工程实践标准+质量指标 | 如 Clean Code/SOLID/TDD，附测试覆盖率 |
| 引用清单 | 表格：来源\|版本\|关联内容\|验证方式 | 每条引用必须可独立验证 |

### 最佳实践
- 每个声明用 checklist `[ ]` 格式，便于评审逐项核实
- 引用必须是公开可访问的资源
- 引用清单的"验证方式"列必须具体（如"见 GitHub Issue #42"）

### 常见错误
- ❌ 引用不可验证的来源（内部文档、口头引用）
- ❌ 声称符合标准但无具体条款
- ❌ 引用清单缺版本或关联内容
- ❌ 无工程可信度声明

## 8. bundle_meta.json（项目元数据）

### 用途
机器可读的项目元数据，用于自动化校验和状态追踪。

### 格式
JSON 文件（非 Markdown）。

### 必备字段

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| project_name | string | 项目名称 | `"BOS-FS"` |
| version | string | 语义化版本号 | `"1.0.0"` |
| build_date | string | ISO 8601 构建日期 | `"2026-05-24"` |
| components | array | 实际生成的组件文件列表 | `["README.md", "FAQ.md", ...]` |
| goal.persona | string | 目标用户画像 | `"开发者"` |
| goal.problem | string | 解决的问题 | `"多模型切换成本高"` |
| goal.solution | string | 解决方案 | `"AI 工作流系统"` |
| goal.outcome | string | 预期成果 | `"提升开发效率 40%"` |
| status | string | 构建状态 | `"complete" / "partial" / "error"` |

### 可选字段

| 字段 | 说明 |
|------|------|
| sanitized | 是否已移除敏感信息 (`true/false`) |
| languages | 语言列表 (`["en", "zh"]`) |
| notes | 构建备注 |

### 最佳实践
- `components` 数组必须与实际生成文件严格一致
- `status` 根据 Quality Gates 自检结果填写
- 日期使用 UTC 时区

### 常见错误
- ❌ `components` 列表与实际文件数量不匹配
- ❌ `goal` 字段缺失或为空
- ❌ 版本号格式非语义化
- ❌ `status` 为 `complete` 但实际缺失组件

---

## 全局一致性规则

构建提交包时必须确保：

1. **项目名称** — 所有 8 个组件中的项目名称完全一致
2. **版本号** — 全局统一，`bundle_meta.json` 为权威来源
3. **价值主张** — README、introduction、FAQ 中的价值描述统一
4. **指标真实** — 所有量化指标必须有测量方法和数据来源
5. **风险不可省略** — risk_disclosure.md 必须存在且非空

## 状态判定规则

| 条件 | status |
|------|--------|
| 8 项中 ≥ 6 项完成 | `complete` |
| 8 项中 3-5 项完成 | `partial` |
| 8 项中 < 3 项完成 | `error` |
